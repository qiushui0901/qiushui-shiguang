# 「拾光」项目 STAR 深挖面试问题与回答稿

> 用法：不要逐字背。每个问题先记住“结论句”，再按 STAR 展开：  
> S（Situation，背景）说明业务场景和约束；T（Task，任务）说明你要解决什么；A（Action，行动）说明方案、取舍和细节；R（Result，结果）说明指标、收益和残留风险。

---

## 一、回答问题的通用结构

### 1. 推荐回答节奏

一个技术问题建议按 4 层回答：

1. **先给结论**：我用了什么方案，解决什么问题。
2. **补业务背景**：为什么需要这么做，原来的瓶颈在哪里。
3. **讲关键设计**：数据结构、链路、失败处理、幂等、一致性、性能取舍。
4. **讲结果和边界**：收益指标、风险、如果上生产还要补什么。

示例：

> 这个问题我当时的核心思路是把高频写从 MySQL 前移到 Redis，用 MQ 做异步削峰，MySQL 保留最终事实数据。这样接口可以快速返回，但代价是链路变成最终一致，所以我重点补了幂等、顺序消费、失败重试和低峰对账。

### 2. 遇到不会的问题怎么答

不要硬编。可以这样说：

> 这个点我项目里没有完整落地，但我当时考虑过生产化方案。当前实现能覆盖单机/模拟压测场景，如果上生产，我会补三块：第一是观测指标，第二是失败补偿，第三是容量和限流策略。

### 3. 项目包装要注意

简历里写了比较强的词，比如“高并发”“一致性保障”“推荐覆盖率”“Agent 评估”。面试官会重点验证这些数字和方案是否真实。所以回答时要主动说清楚：

- **测试环境**：本机、Docker、单机服务、模拟数据，而不是线上百万 QPS。
- **一致性级别**：最终一致，不要说强一致。
- **指标口径**：平均值、P95、P99、样本量、数据分布。
- **方案边界**：当前版本能解决什么，生产还缺什么。

# 二、项目总览类

## Q1：你这个项目整体是什么？你负责了哪些核心部分？

### 考察点

面试官想确认你是不是只会罗列技术栈，还是能从业务、架构、职责和结果讲清楚项目。

### STAR 回答

**S：背景**  
这个项目是一个面向泛生活内容社区的后端系统，业务上类似小红书，核心场景包括笔记发布、点赞、收藏、关注、评论、搜索、推荐和 AI 辅助创作。内容社区的特点是读多写多：热点笔记详情会被频繁访问，点赞收藏这类互动写入也非常高频，而且搜索、推荐、计数这些模块都依赖异步数据同步。

**T：任务**  
我主要负责的方向是核心业务链路的后端设计和性能优化，重点包括：  

- 点赞、收藏、关注的高并发写入链路；
- Caffeine + Redis 多级缓存；
- 笔记详情接口并行聚合优化；
- 推荐系统三路召回；
- 基于 Spring AI 的 Agent 编排和工具调用；
- 基于 XXL-Job 的异步数据对账。

**A：行动**  
架构上我把系统拆成了 Gateway、Auth、User、Note、Comment、Count、Relation、Search、Recommend、KV、OSS、ID Generator、Data Align、Agent 等模块。拆分依据不是单纯为了微服务数量，而是按领域职责和数据归属拆：内容元数据归 Note，正文归 KV，计数归 Count，搜索索引归 Search，推荐候选归 Recommend。  

在高频互动场景里，我没有直接同步写 MySQL，而是采用 Redis 预写状态 + RocketMQ 异步落库。Redis Lua 保证状态判断和更新的原子性，MQ 做削峰和最终落库，消费者侧通过幂等、顺序消费、失败重试和对账保证最终一致。  

在读链路上，我对热点笔记详情做 Caffeine + Redis 缓存，同时把用户信息、正文内容等 RPC 调用用 CompletableFuture 并行化，降低接口 RT。

**R：结果**  
在模拟 benchmark 中，笔记详情接口从串行约 187ms 降到并行约 62ms。推荐模块在 5w 笔记、1w 用户模拟数据集上，通过热度、关注、话题三路召回降低了不同用户 Feed 的重复度。  

需要补充的是，这个项目目前更偏工程实践和模拟压测，不是完整生产系统。如果真要上生产，我会继续补链路追踪、Prometheus 指标、DLQ 死信处理、自动化扩缩容和更完整的灰度发布。

### 追问补充

如果面试官问“你觉得最难的是什么”，可以回答：

> 最难的不是某个中间件用法，而是异步链路的一致性边界。Redis、MQ、MySQL、ES 都有自己的状态，如果只说“用了 MQ”是不够的，必须把失败矩阵、幂等和对账设计完整。

## Q2：为什么拆这么多微服务？如果重新设计，你会怎么拆？

### 考察点

考察你是否理解微服务拆分的收益和代价，而不是“服务越多越高级”。

### STAR 回答

**S：背景**  
项目最初是为了模拟内容社区的完整后端能力，所以把认证、网关、用户、笔记、评论、计数、关注、搜索、推荐、对象存储、KV、AI Agent 等都拆成了独立模块。

**T：任务**  
拆分时要解决两个问题：第一是模块职责清晰，第二是高频模块可以独立优化。例如 Count 服务可以针对计数读写做 Redis Hash 和 MQ 消费优化，Search 服务可以单独处理 ES 索引同步。

**A：行动**  
我的拆分原则是：

- **按领域边界拆**：User 管用户资料，Note 管笔记元数据，KV 管正文内容，Count 管计数，Relation 管关注关系。
- **按技术特性拆**：Search 依赖 ES，Recommend 依赖 ES + Redis 兴趣画像，Agent 依赖 Spring AI 和模型调用。
- **按扩展性拆**：点赞、计数、搜索、推荐这些负载特征不同，独立部署更容易针对性扩容。

但拆得太细也有问题：链路变长、运维成本变高、分布式事务复杂、开发调试成本上升。比如一个笔记详情可能要经过 Note、User、KV、Count 多个服务，任何一个服务慢都会拖慢整体。

**R：结果**  
当前拆分适合作为微服务训练项目，能覆盖完整链路。但如果是小团队真实上线，我会收敛服务数量：  

- User 和 Relation 可以先合并；
- Note 和 Comment 可以先合并为内容域；
- Count 和 Data Align 可以放在同一域；
- Search、Recommend、OSS、Gateway、Auth、ID Generator 继续独立。

这样可以在保留核心扩展性的同时降低系统复杂度。

### 追问补充

可以主动补一句：

> 微服务拆分的关键不是数量，而是服务边界内的数据是否高内聚、服务之间是否低耦合。如果拆分后大量接口强同步互相调用，那其实只是把单体复杂度变成了分布式复杂度。

# 三、高并发互动写入与一致性

## Q3：为什么点赞、收藏、关注不用直接写 MySQL，而用 Redis 预写 + MQ？

### 考察点

考察高并发写入、削峰、最终一致性和系统取舍。

### STAR 回答

**S：背景**  
内容社区里点赞、收藏、关注是典型高频互动操作。如果每次点击都同步写 MySQL，热点笔记或大 V 用户场景下会出现大量并发 insert/update，数据库会承受很高的写压力，而且接口 RT 也容易被数据库锁等待和 IO 拖慢。

**T：任务**  
我要解决两个问题：第一，用户操作要快速返回；第二，数据库最终要有可靠的关系记录和计数结果。

**A：行动**  
我采用了 Redis 预写 + RocketMQ 异步落库：

1. 请求到达后，先通过 Redis Lua 脚本判断当前用户是否已经点赞/收藏/关注。
2. Lua 脚本在 Redis 内完成状态判断和原子更新，避免并发下先查后写造成竞态。
3. Redis 更新成功后，发送 RocketMQ 消息。
4. 消费者异步落库，关系表通过唯一索引或 update 语义保证幂等。
5. 计数更新再通过 MQ 解耦给 Count 服务。
6. 低峰期用 XXL-Job 做增量对账，修复 Redis、MySQL、计数之间的不一致。

**R：结果**  
这样接口可以把同步链路缩短到 Redis + MQ 发送，数据库写入被削峰到消费者侧批量处理。代价是系统从强一致变成最终一致，所以我重点补了顺序消息、幂等消费、失败重试和对账。

### 正确边界

不能说：

> 这个方案保证强一致。

应该说：

> 这个方案保证的是业务可接受范围内的最终一致。用户刚操作后的展示优先读 Redis 状态，MySQL 用异步链路追平。

## Q4：Redis 写成功，但 MQ 发送失败怎么办？

### 考察点

这是核心追问。面试官想看你是否真的理解异步链路失败矩阵。

### STAR 回答

**S：背景**  
Redis 预写 + MQ 异步落库最大的风险是中间状态断裂。Redis 是用户实时状态，MySQL 是最终事实数据，如果 Redis 成功但 MQ 没发出去，MySQL 就不会被更新。

**T：任务**  
要保证这类异常不会永久丢失，至少要有可发现、可补偿机制。

**A：行动**  
当前项目里基础方案是接口侧捕获 MQ 发送异常并记录日志，低峰期通过数据对账补偿。但如果上生产，我会进一步增强为三层机制：

1. **本地操作流水表**  
   Redis Lua 成功后，先记录一条操作流水，包含 userId、bizId、action、eventTime、status。MQ 发送成功后标记为 SENT。  

2. **可靠消息补偿任务**  
   定时扫描状态为 INIT 或 SEND_FAILED 的流水，重新投递 MQ。这样 MQ 发送失败不会导致事件永久丢失。  

3. **最终对账**  
   低峰期用 XXL-Job 按 userId 或 noteId 分片，对比 Redis 状态、MySQL 关系表和计数表，发现差异后修复。

**R：结果**  
项目当前能通过对账兜底最终一致，但生产级方案最好引入操作流水或 outbox 模式，让“Redis 成功、MQ 失败”这类异常可追踪、可重放。

### 追问补充

如果问“为什么不直接用事务消息”，可以答：

> RocketMQ 事务消息能保证本地事务和消息发送之间的一致性，但这里的本地事务如果是 Redis Lua，不是传统数据库事务，事务消息并不能天然覆盖 Redis 状态和 MQ 的原子一致。要更稳，还是需要操作流水或 outbox。

## Q5：点赞后马上取消点赞，怎么保证最终状态正确？

### 考察点

考察顺序消息、状态覆盖、幂等和乱序处理。

### STAR 回答

**S：背景**  
用户连续操作是互动系统的典型场景，比如先点赞，再取消，再点赞。如果消息乱序消费，MySQL 最终状态可能和用户最后一次操作相反。

**T：任务**  
目标是保证同一业务对象上的连续操作最终按真实发生顺序落库。

**A：行动**  
我的处理思路分三层：

1. **Redis 侧原子判断**  
   Lua 脚本先判断当前状态，重复点赞或重复取消直接返回业务状态，不产生无意义事件。

2. **MQ 顺序消息**  
   对同一用户和同一笔记的操作，用相同 sharding key 投递到同一个队列，例如 `userId + noteId`。这样点赞、取消、再点赞可以在同一队列里按顺序消费。

3. **数据库侧状态更新幂等**  
   点赞表使用 userId + noteId 唯一索引。点赞用 insert or update status=liked，取消用 update status=unliked。重复消费不会新增多条记录。

**R：结果**  
这样可以保证单个用户对单个笔记的连续操作最终状态正确。  
不过需要注意：顺序粒度不能选得太粗。如果只按 userId，会让这个用户的所有点赞串行；如果只按 noteId，爆款笔记会集中到一个队列。更合理的是按业务关系 key，比如 `userId-noteId`。

### 追问补充

如果问“RocketMQ 顺序消费失败会怎样”，可以答：

> 顺序消费失败会阻塞该队列后续消息，保证顺序的代价就是局部吞吐下降。所以消费者逻辑要尽量轻量，并通过限流、重试、死信和告警避免长时间卡队列。

## Q6：你怎么做消息幂等？

### 考察点

考察重复消费、数据库约束和业务语义。

### STAR 回答

**S：背景**  
MQ 至少一次投递是常态，消费者可能因为超时、异常、重平衡等原因重复消费。同一条点赞消息可能被消费多次。

**T：任务**  
我要保证重复消费不会造成重复关系、计数多加或状态回退。

**A：行动**  
幂等分两类处理：

1. **关系表幂等**  
   点赞、收藏、关注关系表用联合唯一索引，比如 `(user_id, note_id)` 或 `(user_id, target_user_id)`。点赞消息用 insert or update，取消点赞用按唯一键更新状态。重复消费同一条消息不会产生多条关系。

2. **计数幂等**  
   计数比关系表更容易出错，因为重复消费可能导致 like_total 多加。生产级做法是给每个事件一个 eventId，计数消费者先判断 eventId 是否处理过，再执行 incr/decr。  
   当前项目更偏模拟实现，基础幂等主要依赖关系表状态和对账修复。如果上生产，我会补事件去重表或 Redis 去重 key。

**R：结果**  
关系数据可以依靠唯一索引保证幂等，计数数据需要额外事件幂等或周期性对账。回答时要区分这两类，不能笼统说“唯一索引就够了”。

## Q7：为什么用 Lua 脚本？它解决了什么，解决不了什么？

### 考察点

考察 Redis 原子性边界。

### STAR 回答

**S：背景**  
点赞这类操作如果在应用层写成“先查 Redis，再判断，再写 Redis”，并发下会有竞态。例如两个请求同时判断未点赞，然后都写入，导致重复事件。

**T：任务**  
需要把状态判断和状态更新合成一个原子操作。

**A：行动**  
我把判断和更新放到 Redis Lua 脚本里执行。Redis 执行 Lua 时是单线程原子执行的，中间不会被其他命令打断。脚本可以完成：

- 判断用户是否已点赞；
- 对 Bitmap/ZSet/Hash 做状态更新；
- 设置 TTL；
- 返回明确的业务状态码，比如成功、已点赞、不存在等。

**R：结果**  
Lua 解决的是 Redis 内部多命令原子性问题，避免并发状态错乱。  
但 Lua 解决不了 Redis 和 MQ、MySQL 之间的分布式事务问题。所以后面还要靠 MQ 重试、操作流水、消费者幂等和对账来保证最终一致。

# 四、Buffer Trigger 与批量落库

## Q8：消费者侧为什么要做 Buffer Trigger 聚合批量写入？

### 考察点

考察削峰、批处理、延迟和数据安全。

### STAR 回答

**S：背景**  
如果消费者每收到一条点赞消息就写一次数据库，高峰期数据库会被大量小事务打满。尤其是点赞、收藏、计数更新这类写入，单条 SQL 成本不高，但请求量很大。

**T：任务**  
目标是降低数据库 QPS 和事务提交次数，同时控制用户可感知延迟。

**A：行动**  
我的思路是在消费者侧用 Buffer Trigger 做聚合：

- 按条数触发，比如攒够 500 条批量写；
- 按时间触发，比如 1 秒内没攒够也强制 flush；
- 对同一业务 key 可以做合并，比如同一用户对同一笔记多次操作，只保留最后状态；
- 批量写库时使用 batch insert 或批量 update，减少网络往返和事务提交。

**R：结果**  
数据库写入从大量单条操作变成批量操作，能降低高峰期写入压力。代价是落库延迟增加，并且 buffer 在内存里有丢失风险。生产级方案要结合 MQ 的消费确认：只有批量写库成功后才 ack；失败则不确认，让 MQ 重试。

### 追问补充

如果问“服务重启 buffer 丢了怎么办”，可以答：

> 正确做法是消息没有成功落库前不 ack，这样服务重启后 MQ 会重新投递。内存 buffer 只是消费过程中的临时聚合，不应该成为唯一可靠存储。

# 五、多级缓存

## Q9：为什么使用 Caffeine + Redis 二级缓存？

### 考察点

考察本地缓存、分布式缓存和一致性取舍。

### STAR 回答

**S：背景**  
笔记详情、用户信息、计数这类数据读多写少，热点内容会被频繁访问。如果每次都查 Redis，虽然比 MySQL 快，但仍然有网络开销；如果只查 MySQL，会扛不住热点流量。

**T：任务**  
希望降低接口 RT，同时降低 Redis 和 MySQL 压力。

**A：行动**  
我采用了 Caffeine + Redis + MySQL 的多级读取：

1. 先查 Caffeine 本地缓存，命中后无网络开销；
2. 本地缓存 miss 后查 Redis；
3. Redis miss 后查 MySQL 或 RPC 聚合；
4. 查到结果后异步回填 Redis；
5. Redis 命中后异步回填 Caffeine；
6. 对缓存设置 TTL 和随机过期，避免雪崩；
7. 对不存在的数据缓存短 TTL 空值，缓解穿透。

**R：结果**  
本地缓存能明显降低热点详情的访问延迟，Redis 作为分布式缓存承接跨实例共享，MySQL 作为最终数据源。  
但二级缓存最大问题是多实例一致性，所以更新笔记、删除笔记、权限变化时必须删除 Redis，并通过 MQ 或广播通知删除本地 Caffeine。

---

## Q10：本地缓存和 Redis 不一致怎么办？

### 考察点

考察缓存一致性边界。

### STAR 回答

**S：背景**  
Caffeine 是每个服务实例自己的本地缓存。Redis 更新或删除后，其他实例的本地缓存不会自动失效，可能继续返回旧数据。

**T：任务**  
要尽量缩短本地缓存和 Redis/DB 的不一致窗口。

**A：行动**  
我采用的策略是：

- 写操作先删除 Redis 缓存；
- 更新数据库；
- 发送延迟删除消息，做延迟双删；
- 通过 MQ 广播或服务内事件通知删除各实例 Caffeine；
- 本地缓存设置较短 TTL，比如 1 小时，作为最终兜底。

**R：结果**  
这个方案不能保证强一致，但能降低旧数据长期存在的概率。生产场景如果对权限敏感，比如私密笔记改公开或公开改私密，我会让权限判断不完全依赖缓存中的 VO，而是在返回前重新校验权限字段或让权限变更强制广播失效。

### 重要提醒

面试官可能会问：

> 如果私密笔记被缓存了，别人会不会看到？

回答重点：

> 缓存可以缓存内容，但返回前必须做用户身份和可见性校验。缓存不是权限边界，权限校验必须在业务层兜底。

---

## Q11：延迟双删为什么只能保证最终一致？

### 考察点

考察缓存更新经典问题。

### STAR 回答

**S：背景**  
缓存和数据库更新不是一个原子事务。典型问题是：线程 A 删除缓存后准备更新 DB，线程 B 此时缓存 miss，读到旧 DB 并写回缓存，之后线程 A 才完成 DB 更新，导致缓存旧值。

**T：任务**  
要清理并发读写期间被回填的旧缓存。

**A：行动**  
延迟双删的做法是：

1. 第一次删除缓存；
2. 更新数据库；
3. 延迟一段时间后第二次删除缓存。

延迟时间要大于一次“读 DB + 写缓存”的 P99 耗时，并加一定安全余量。项目里可以通过 RocketMQ 延迟消息实现，而不是让业务线程 sleep。

**R：结果**  
第二次删除能清掉并发读回填的旧缓存。但它仍然不是强一致，因为第二次删除消息可能失败，或者极端慢查询超过延迟窗口。它保证的是最终一致，最后还依赖 TTL 和对账兜底。

# 六、CompletableFuture 接口优化

## Q12：你说接口 RT 从 187ms 降到 62ms，怎么优化的？

### 考察点

考察是否真的理解并行化收益和指标口径。

### STAR 回答

**S：背景**  
笔记详情接口需要聚合多类数据：笔记元数据、作者信息、正文内容、互动状态、计数信息。最初如果串行调用，整体耗时接近所有依赖耗时之和。

**T：任务**  
目标是在不改变业务结果的前提下，降低接口响应时间。

**A：行动**  
我分析后发现有些依赖没有强顺序关系。例如拿到 noteDO 后，用户信息和正文内容可以并行查询；互动状态和计数也可以并行聚合。于是用 CompletableFuture 把多个 RPC 查询提交到业务线程池：

- `userRpcService.findById` 查询作者信息；
- `keyValueRpcService.findNoteContent` 查询正文；
- `countRpcService.findByNoteIds` 查询计数；
- 点赞/收藏状态从 Redis Bitmap/Roaring Bitmap 判断。

最后通过 `CompletableFuture.allOf()` 汇总结果。

**R：结果**  
在 benchmark 中，串行平均 RT 约 187ms，并行后约 62ms，接近最慢的单个依赖耗时。  
这个优化的边界是：并行会增加瞬时下游压力，所以必须配合线程池隔离、超时控制、降级和限流。

### 追问补充

如果问“线程池怎么设置”，可以答：

> 线程池不能直接用 ForkJoinPool.commonPool。我会用独立业务线程池，核心线程数按 CPU 和 IO 等待比例估算，设置有界队列和拒绝策略，并监控 activeCount、queueSize、rejectCount。RPC 都要设置超时，避免一个下游慢调用拖住整个接口。

## Q13：CompletableFuture 里的 userId 怎么传递？

### 考察点

考察 ThreadLocal、TTL、异步上下文。

### STAR 回答

**S：背景**  
登录用户 userId 通常存在 ThreadLocal 中。请求线程能拿到，但 CompletableFuture 在线程池里执行时，普通 ThreadLocal 不会自动传递。

**T：任务**  
异步任务里如果还需要 userId，就必须解决上下文传递问题。

**A：行动**  
可选方案有两种：

1. **显式传参**  
   最稳妥，把 userId 作为方法参数传给异步任务，避免隐式上下文。

2. **TransmittableThreadLocal**  
   如果框架里大量依赖 ThreadLocal，可以用 TTL 包装线程池任务，在提交任务时捕获父线程上下文，在子线程执行前恢复，执行后清理。

**R：结果**  
项目里更推荐核心链路显式传参，框架通用能力可以用 TTL。需要注意 TTL 不适用于响应式 WebClient/Reactor 场景，响应式链路应该用 Reactor Context。

# 七、RocketMQ

## Q14：你项目里用了哪些 RocketMQ 消息类型？

### 考察点

考察 MQ 基础能力和项目映射。

### STAR 回答

**S：背景**  
项目里不同业务场景对消息的要求不同，有的需要解耦，有的需要顺序，有的需要延迟，有的需要和本地事务协调。

**T：任务**  
要根据场景选择消息类型。

**A：行动**  
项目里主要用了几类：

- **普通消息**：笔记发布后通知计数、搜索、推荐等模块；
- **顺序消息**：点赞、取消点赞、关注、取关这类连续操作，保证同一业务 key 的消费顺序；
- **延迟消息**：用于延迟双删，比如更新笔记后延迟删除 Redis 缓存；
- **事务消息**：发布带正文的笔记时，协调 Note 元数据和 KV 正文存储链路。

**R：结果**  
MQ 帮项目完成了解耦、削峰和最终一致。但每类消息都有代价：顺序消息牺牲局部吞吐，延迟消息不是精确定时，事务消息也不能解决所有跨资源一致性问题。

---

## Q15：发布笔记为什么用事务消息？

### 考察点

考察 RocketMQ 事务消息和本地事务边界。

### STAR 回答

**S：背景**  
发布笔记有两个核心动作：保存笔记元数据和保存正文内容。正文内容可能通过 KV 服务异步保存，如果其中一个成功、另一个失败，会出现不完整笔记。

**T：任务**  
希望尽量保证发布事件和本地状态一致，避免消息发出但本地事务失败，或者本地成功但消息丢失。

**A：行动**  
我使用 RocketMQ 事务消息。流程是：

1. 先发送 half message；
2. 执行本地事务，例如保存 Note 元数据；
3. 本地事务成功则 commit 消息；
4. 本地事务失败则 rollback；
5. 如果生产者异常，RocketMQ 会回查本地事务状态。

**R：结果**  
事务消息可以保证“本地事务”和“消息可见性”的一致性。  
但它不能天然保证多个远程服务之间的强一致，比如 Note 服务和 KV 服务之间仍然需要幂等、重试和补偿。

# 八、MySQL 与数据模型

## Q16：点赞表为什么用唯一索引？取消点赞为什么不直接删除？

### 考察点

考察数据建模、幂等和审计。

### STAR 回答

**S：背景**  
点赞关系是用户和笔记之间的二元关系，同一个用户对同一笔记只能有一条有效记录。

**T：任务**  
需要防止重复数据，并支持 MQ 重复消费下的幂等。

**A：行动**  
点赞表用 `(user_id, note_id)` 做联合唯一索引。点赞时 insert，如果已存在则 update status=liked；取消点赞时 update status=unliked，而不是物理删除。

**R：结果**  
这样有几个好处：

- 重复点赞不会插入多条数据；
- MQ 重复消费可以通过 update 幂等处理；
- 保留用户历史行为，后续推荐画像可以利用；
- 对账时能知道这个关系曾经存在过。

缺点是表会持续增长，所以长期要配合冷热分离、分区或归档。

---

## Q17：计数为什么不直接实时 update MySQL？

### 考察点

考察热点行更新和计数一致性。

### STAR 回答

**S：背景**  
笔记点赞数、收藏数、评论数是热点计数。如果爆款笔记每次点赞都同步 `update note_count set like_total=like_total+1`，同一行会被大量并发更新，容易出现锁竞争。

**T：任务**  
需要提升写入吞吐，同时让用户看到的计数尽量实时。

**A：行动**  
我把计数分成两层：

- Redis Hash 维护实时计数，用户读接口优先读 Redis；
- MySQL Count 表异步落库，作为持久化基准；
- 点赞/取消点赞事件通过 MQ 通知 Count 服务增减；
- 对账任务定期修复 Redis 和 MySQL 计数差异。

**R：结果**  
这样能避免热点行频繁同步更新 MySQL。代价是计数是最终一致，短时间内 MySQL 可能落后 Redis。用户体验上优先展示 Redis 计数即可。

# 九、Redis 数据结构

## Q18：为什么使用 Roaring Bitmap 判断点赞/收藏状态？

### 考察点

考察 Redis 数据结构选型。

### STAR 回答

**S：背景**  
笔记列表和详情页都需要判断当前用户是否点赞、是否收藏。如果每篇笔记都查 MySQL，会产生大量查询。

**T：任务**  
需要快速判断“某用户是否对某笔记有过互动”，并尽量节省内存。

**A：行动**  
我使用用户维度的 Roaring Bitmap 缓存点赞/收藏集合。判断时用 noteId 做元素，查询复杂度接近 O(1)。相比普通 Bitmap，Roaring Bitmap 对稀疏 ID 更友好，能压缩存储。

**R：结果**  
查询点赞状态时可以优先走 Redis，减少 MySQL 压力。  
但这里有一个边界：如果 noteId 是雪花 ID，数值空间非常大，普通 Bitmap 不合适；Roaring Bitmap 虽然能处理稀疏集合，但实际落地要确认 Redis 模块支持和序列化方式。生产里也可以用 ZSet 或 Set 做替代，按用户最近点赞列表维护。

---

## Q19：ZSet、Set、Bitmap 分别适合哪些互动场景？

### STAR 回答

**S：背景**  
点赞、收藏、关注既有“是否存在”的判断，也有“按时间排序列表”的查询。

**T：任务**  
需要根据查询模式选择数据结构。

**A：行动**  
我的选择是：

- **Set**：适合只判断是否存在，比如某用户是否关注某人。
- **ZSet**：适合需要按时间排序，比如用户点赞列表、收藏列表、关注列表，score 用操作时间。
- **Bitmap / Roaring Bitmap**：适合大规模状态判断，查询快、内存省，但不适合直接做最近列表。
- **Hash**：适合聚合多个计数字段，比如 note_count:{noteId} 里存 like_total、collect_total、comment_total。

**R：结果**  
选型核心不是哪个结构高级，而是查询模式。需要排序就用 ZSet，只判断存在可以 Set 或 Bitmap，多个字段聚合读写用 Hash。

# 十、推荐系统

## Q20：为什么设计热度、关注、话题三路召回？

### 考察点

考察推荐系统基本理解。

### STAR 回答

**S：背景**  
如果只做热度推荐，所有用户看到的内容会高度相似，容易变成全站热门榜，个性化很弱。内容社区需要同时兼顾热门内容、社交关系和用户兴趣。

**T：任务**  
目标是提高推荐覆盖率，降低不同用户 Feed 重叠率。

**A：行动**  
我设计了三路召回：

1. **热度召回**  
   用 ES `function_score` 综合点赞、收藏、评论和时间衰减，保证基础内容质量。

2. **关注召回**  
   从 Redis ZSet 获取用户关注的人，再查这些作者的最新笔记，增强社交关系。

3. **话题召回**  
   根据用户点赞、收藏行为更新 Redis ZSet 里的话题兴趣权重，再召回相关话题内容。

三路结果合并时做去重，并按关注、话题、热度做优先级融合。

**R：结果**  
在 5w 笔记、1w 用户模拟数据集上，相比单路热度召回，三路召回能显著降低不同用户 Feed 的重叠。这个指标是离线模拟结果，不等同于线上 CTR 提升。

---

## Q21：`function_score` 的权重怎么定？

### STAR 回答

**S：背景**  
热度排序不能只看点赞数，否则老内容会长期霸榜；也不能只看时间，否则质量差的新内容会排太高。

**T：任务**  
需要同时考虑互动质量和时间衰减。

**A：行动**  
我用 ES `function_score` 组合多个因子：

- 点赞数权重 0.5；
- 收藏数权重 0.3；
- 评论数权重 0.2；
- 对计数字段用 sqrt modifier，避免头部内容分数过大；
- 用 gauss decay 对发布时间做 7 天时间衰减。

权重初始值来自业务直觉：点赞最轻量但最多，收藏代表更强兴趣，评论代表互动深度。真实生产中这些权重应该通过离线评估和 A/B 实验迭代，而不是固定写死。

**R：结果**  
这个方案能让新内容有一定曝光机会，同时避免互动数巨大的老内容长期占据前排。

---

## Q22：推荐里的冷启动怎么解决？

### STAR 回答

**S：背景**  
新用户没有关注关系，也没有点赞收藏行为，关注召回和话题召回都会为空。

**T：任务**  
需要在缺少个性化数据时仍然给出可接受的 Feed。

**A：行动**  
我会分用户冷启动和内容冷启动处理：

- 新用户：先使用热度召回、地域/时间新鲜度、全站热门话题；
- 新内容：给新发布内容一定探索流量，不能完全依赖点赞数；
- 新兴趣：用户一旦点赞、收藏、关注，就快速更新 Redis 兴趣画像；
- 长期优化：引入标签、点击停留时长和负反馈。

**R：结果**  
当前项目主要实现了基础三路召回，冷启动还比较简单。如果生产化，需要增加探索机制和在线反馈闭环。

# 十一、Elasticsearch 与搜索同步

## Q23：MySQL 和 ES 怎么保持一致？

### 考察点

考察搜索索引同步和最终一致性。

### STAR 回答

**S：背景**  
MySQL 是主数据源，ES 是搜索索引。笔记发布、更新、删除后，如果 ES 不同步，搜索结果就会出现旧数据或缺数据。

**T：任务**  
需要保证 ES 索引最终追上 MySQL。

**A：行动**  
常见方案有两种：

1. **业务 MQ 同步**  
   笔记发布或更新后发送 MQ，Search 服务消费消息更新 ES。

2. **Canal 监听 binlog**  
   Canal 监听 MySQL binlog，把变更同步给 ES 或发送到 MQ。

项目里简历写了 Canal 和 MQ，推荐回答是：核心业务事件通过 MQ 主动同步，Canal 可以作为数据库变更监听和补偿链路。同步失败时记录失败日志，进入重试或补偿任务。

**R：结果**  
ES 同步是最终一致，不应该承诺实时强一致。搜索短时间查不到刚发布内容是可接受的，但删除或私密内容需要更强兜底，比如搜索结果返回前二次校验状态和可见性。

# 十二、AI Agent

## Q24：你项目里的 Agent 是怎么设计的？

### 考察点

考察你是否真的理解 Agent、Tool Calling、Orchestrator，而不是只写热门词。

### STAR 回答

**S：背景**  
内容社区里 AI 能做的事情很多，比如辅助创作、内容理解、内容审核、搜索总结。如果所有能力都塞进一个大 prompt，会难维护、难评估。

**T：任务**  
我希望把不同智能能力拆成职责明确的 Agent，并通过编排器控制执行流程。

**A：行动**  
我设计了几个专项 Agent：

- ContentUnderstandingAgent：理解用户输入和内容意图；
- ContentModerationAgent：判断内容是否安全；
- CreativeAssistantAgent：生成创作建议或正文；
- SearchSummarizeAgent：结合搜索结果做总结。

AgentOrchestrator 支持串行、并行和条件执行。比如流式创作场景里，先执行理解和审核，如果审核不通过，就直接返回拒绝原因；如果通过，再让创作助手流式输出。  
Tool Calling 接入 ES 搜索、Redis 热度数据和敏感词检测，让模型不只靠自身知识回答。

**R：结果**  
这种拆分让每个 Agent 的输入输出更清楚，也方便做 L1 冒烟测试和 L2 Golden Dataset 评估。简历里的 94.3% 指的是模拟评估集中路由、工具调用和输出格式符合预期的比例。

## Q25：Agent 的 Golden Dataset 怎么评估？

### 考察点

考察 AI 工程化能力。

### STAR 回答

**S：背景**  
Agent 输出不稳定，如果只靠人工试几条样例，很难保证改 prompt 或改工具后没有回归。

**T：任务**  
需要构建一套可重复运行的评估集。

**A：行动**  
我把评估分成两层：

- **L1 冒烟测试**：检查接口能否正常返回、JSON 格式是否正确、工具是否能被调用。
- **L2 Golden Dataset**：构造典型用户输入，标注期望路由、是否需要审核、应该调用哪些工具、输出必须包含哪些结构字段。

评分不只看文本相似度，而是看结构化断言，比如：

- 是否调用搜索工具；
- 是否拒绝敏感内容；
- 返回 JSON 是否可解析；
- 是否包含标题、正文、标签等字段；
- 是否没有输出违规内容。

**R：结果**  
L2 通过率 94.3% 表示大部分样本在路由、工具调用和格式上符合预期。失败样本主要来自模型输出格式漂移或边界意图识别错误。后续可以通过更严格的结构化输出、函数调用 schema 和重试修复。

## Q26：怎么防 Prompt 注入？

### STAR 回答

**S：背景**  
Agent 接入工具后，用户可能通过 prompt 注入诱导模型忽略规则、泄露敏感词库、越权调用工具。

**T：任务**  
需要保证模型不能决定所有安全边界。

**A：行动**  
我的设计原则是：权限控制不能只靠 prompt。

- 工具层做白名单，只暴露必要工具；
- 工具参数做校验，比如只能查公开笔记；
- 用户身份从后端上下文取，不允许模型自己传 userId；
- 敏感操作需要服务端权限判断；
- 输出前再经过内容审核；
- 对系统 prompt、工具返回和用户输入做分层隔离。

**R：结果**  
Prompt 可以提升模型遵循度，但不能作为安全机制。真正的权限边界必须在后端代码和工具层实现。

# 十三、网关与认证

## Q27：Gateway 传 userId 给下游安全吗？

### 考察点

考察内部服务安全。

### STAR 回答

**S：背景**  
Gateway 鉴权后把 userId 放到 Header 传给下游服务，这是微服务里常见做法。但如果有人绕过网关直接请求内部服务，并伪造 Header，就会有安全风险。

**T：任务**  
需要保证下游服务不能盲目信任外部传入的 Header。

**A：行动**  
当前项目主要是训练项目，默认内部服务在内网中，不直接暴露公网。生产级我会补几层防护：

- 网络隔离：内部服务只在 VPC 或 K8s ClusterIP 暴露；
- NetworkPolicy：只允许 Gateway 或指定服务访问；
- 内部调用签名：Gateway 或服务间调用加 HMAC 签名和时间戳；
- 下游 Filter 校验内部签名；
- 重要接口二次鉴权。

**R：结果**  
userId Header 传递本身没问题，但它不是安全边界。安全边界应该由网关鉴权、网络隔离和服务间签名共同保证。

# 十四、分布式 ID

## Q28：为什么需要分布式 ID？

### 考察点

考察 ID 生成、数据库扩展和雪花算法。

### STAR 回答

**S：背景**  
笔记、评论、用户行为等数据分布在不同服务里，如果都依赖数据库自增 ID，会和单库强绑定。未来分库分表后，自增 ID 也容易冲突。

**T：任务**  
需要一个全局唯一、趋势递增、性能高的 ID 生成方案。

**A：行动**  
项目里有独立分布式 ID 服务，支持 Snowflake 和 Segment 两类思路：

- Snowflake 适合高性能本地生成，ID 包含时间戳、workerId、序列号；
- Segment 适合从数据库批量取号段，本地内存发号，减少 DB 访问。

**R：结果**  
全局 ID 可以让各业务服务独立生成主键，趋势递增也有利于 MySQL B+Tree 插入性能。需要注意雪花算法要处理 workerId 分配和时钟回拨。

---

## Q29：雪花算法时钟回拨怎么办？

### STAR 回答

**S：背景**  
Snowflake 依赖系统时间。如果机器时间回拨，可能生成重复 ID。

**T：任务**  
需要避免 ID 重复，同时保证服务可用性。

**A：行动**  
常见处理方式：

- 小幅回拨：等待时间追上，比如回拨几毫秒可以 sleep；
- 大幅回拨：拒绝发号并报警；
- 使用 Zookeeper/Redis 记录 worker 节点和最后时间戳；
- 部署层面禁止频繁 NTP 大幅校时。

**R：结果**  
时钟回拨不能忽略。生产系统宁愿短暂不可用，也不能生成重复主键。

# 十五、可观测性与压测

## Q30：你的 benchmark 怎么做的？187ms 到 62ms 是什么口径？

### 考察点

考察指标可信度。

### STAR 回答

**S：背景**  
简历里写性能数据，面试官一定会问测试环境和口径。如果说不清楚，数据可信度会下降。

**T：任务**  
要明确说明这是模拟 benchmark，不是线上真实流量。

**A：行动**  
我会这样回答：

> 这个数据是在本地/测试环境下，用模拟数据对笔记详情接口做的对比测试。优化前是串行调用用户服务、KV 服务和计数服务；优化后用 CompletableFuture 并行聚合。187ms 和 62ms 是同一环境、同一数据规模下的平均响应时间对比。

如果有 P95/P99，可以继续补；如果没有，不要硬说。

**R：结果**  
这个数据能证明并行化方向有效，但不能直接代表生产性能。生产验证还需要 P95/P99、QPS、错误率、线程池指标、下游服务耗时和 GC 指标。

## Q31：线上接口突然变慢，你怎么排查？

### STAR 回答

**S：背景**  
微服务链路长，接口变慢可能来自网关、业务线程池、Redis、MySQL、MQ、下游 RPC、ES 等多个环节。

**T：任务**  
要快速定位瓶颈，而不是盲目改代码。

**A：行动**  
我的排查路径是：

1. 看整体指标：RT、P95/P99、错误率、QPS 是否同时变化；
2. 看链路追踪：定位慢在哪个服务、哪个 RPC；
3. 看线程池：active、queue、reject 是否异常；
4. 看 Redis：命中率、慢查询、大 key、连接池；
5. 看 MySQL：慢 SQL、锁等待、连接数、CPU、IO；
6. 看 MQ：积压、消费失败、重试、DLQ；
7. 看 ES：查询耗时、线程池、segment、GC。

**R：结果**  
先定位再优化。比如如果是 Redis 命中率下降，就看缓存失效；如果是线程池队列堆积，就看下游慢调用或池参数；如果是 MySQL 慢 SQL，就看索引和执行计划。

# 十六、开放式架构题

## Q32：如果 DAU 从 1 万涨到 100 万，你先改哪里？

### STAR 回答

**S：背景**  
用户规模扩大后，瓶颈通常先出现在热点读、高频写、搜索推荐和数据库容量。

**T：任务**  
要分阶段扩展，不能一上来全面重构。

**A：行动**  
我会按优先级处理：

1. **读链路扩展**  
   提高 Redis/Caffeine 命中率，热点 key 保护，接口限流。

2. **写链路削峰**  
   点赞收藏关注全部走 MQ，消费者横向扩容，批量落库。

3. **数据库治理**  
   慢 SQL、索引优化，计数表和关系表按 userId/noteId 分库分表。

4. **搜索推荐独立扩展**  
   ES 集群扩容，推荐召回缓存化，异步更新用户画像。

5. **可观测性补齐**  
   traceId、Prometheus、日志、告警、容量水位。

**R：结果**  
第一阶段先保护核心链路：浏览、详情、点赞、发布。非核心能力比如 AI Agent 可以单独限流和降级，避免拖垮主业务。

---

## Q33：如果点赞必须强一致，你怎么改？

### STAR 回答

**S：背景**  
当前 Redis 预写 + MQ 是最终一致。如果产品要求点赞后 MySQL 立刻一致，就不能只依赖异步落库。

**T：任务**  
需要在一致性和性能之间重新取舍。

**A：行动**  
强一致方案可以改成：

- 接口同步写 MySQL 关系表；
- 同一事务内更新计数或写操作流水；
- Redis 缓存作为读优化，写成功后删除或更新；
- 对热点笔记增加行级锁冲突治理，比如分片计数；
- 接口 RT 增加，吞吐下降，需要限流。

也可以做折中：关系表同步强一致，计数异步最终一致。这样用户是否点赞准确，但点赞数允许短暂延迟。

**R：结果**  
我会优先推荐折中方案，因为用户最敏感的是“我有没有点赞”，而不是全站点赞数精确到毫秒。

## Q34：这个项目最大的技术债是什么？

### STAR 回答

**S：背景**  
项目覆盖面比较广，但作为学习型项目，有些生产级能力还不完整。

**T：任务**  
需要能客观识别技术债，而不是只讲优点。

**A：行动**  
我认为主要技术债有：

**R：结果**  
这些技术债不影响项目作为工程实践展示，但如果生产化，我会优先补可靠消息、观测告警和安全隔离。

# 十七、面试时的高分表达模板

## 1. 讲异步一致性

> 这里我没有把它包装成强一致。这个链路本质上是 Redis 承接实时状态，MQ 做异步削峰，MySQL 做最终事实存储。为了让最终一致可控，我重点做了四件事：Redis Lua 保证状态原子变更，MQ 顺序消息保证同一业务 key 的操作顺序，消费者幂等避免重复消费造成脏数据，最后用 XXL-Job 对账兜底。

## 2. 讲缓存

> 我把缓存分成两层看：Caffeine 解决热点数据的本地访问延迟，Redis 解决跨实例共享和抗 DB 压力。这个方案的难点不是读取，而是失效。本地缓存不是权限边界，所以返回前仍要做可见性校验；更新时删除 Redis，并通过延迟删除和广播尽量缩短不一致窗口。

## 3. 讲性能优化

> 这个优化不是简单把代码改成异步，而是先分析依赖关系。用户信息、正文内容、计数、互动状态之间没有严格串行依赖，所以可以并行聚合。优化后 RT 接近最慢的单个依赖耗时，但代价是瞬时并发会变高，所以需要独立线程池、超时、降级和监控。

## 4. 讲推荐

> 单路热度召回的问题是所有人看到的内容趋同，所以我增加了关注召回和话题召回。热度保证基础质量，关注保证社交关系，话题保证兴趣个性化。当前实现更偏召回层，后续如果生产化，还要补精排、探索、负反馈和 A/B 实验。

## 5. 讲 Agent

> 我没有把所有能力塞进一个大模型调用，而是拆成内容理解、审核、创作、搜索总结几个专项 Agent，再用 Orchestrator 编排。这样每个 Agent 的职责和输出更可控，也方便用 Golden Dataset 对路由、工具调用和输出格式做回归测试。

# 十九、最后的面试提醒

回答时不要堆技术名词。每个问题都按这个顺序说：

> 当时的业务问题是什么 -> 我为什么选这个方案 -> 我具体怎么做 -> 它解决了什么 -> 它还有什么边界。

能主动说边界，反而更可信。尤其是“最终一致”“模拟压测”“生产还需补充”这几个点，要说得坦诚但有方案。
