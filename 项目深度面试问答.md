## 一、微服务架构篇

### Q1：你这个项目拆了15个微服务，拆分的依据是什么？有没有遇到过服务拆得过细带来的问题？如果让你重新拆，你会怎么调整？

**答：**

拆分依据主要遵循**单一职责 + 领域驱动**原则，按业务边界拆分：用户、笔记、评论、关注、计数、KV存储、OSS、搜索、推荐、AI Agent、分布式ID、数据对齐、认证、网关各自独立。

拆得过细带来的问题：
- **运维成本高**：每个服务都要独立部署、监控、日志收集，15个服务意味着15套CI/CD流水线
- **调用链路长**：一个笔记详情请求需要经过 Gateway → Note → User(Feign) → KV(Feign) → Count(Feign)，链路越长延迟越高、出故障概率越大
- **分布式事务复杂**：跨服务的数据一致性保障成本高，比如发布笔记需要同时写 Note 元数据和 KV 内容
- **开发效率低**：简单改动可能涉及多个服务的联调

如果重新拆，可以考虑：
- **合并 Count + DataAlign**：计数和对齐本质是同一领域
- **合并 UserRelation + User**：关注关系是用户域的子域
- **合并 Comment + Note**：评论和笔记是内容域，高频交互
- 保留独立的基础设施服务：Gateway、Auth、OSS、DistributedID、Search

### Q2：Gateway 用 Sa-Token 做鉴权，userId 通过 Header 往下游传。如果有人绕过 Gateway 直接调用内部服务怎么办？你们在内部服务层有做二次鉴权吗？生产环境怎么保证内部服务不暴露？

**答：**

当前项目内部服务**没有做二次鉴权**，下游服务直接从 Header 取 userId 信任使用，这是一个安全隐患。

生产环境的防护措施：
1. **网络隔离**：内部服务只部署在内网 VPC，Gateway 暴露在公网 SLB 后面，内部服务端口不对外
2. **K8s NetworkPolicy**：限制只有 Gateway Pod 能访问内部服务 Pod
3. **服务鉴权**：内部服务间调用携带内部签名（如 HMAC），Gateway 不传此签名，外部请求无法伪造
4. **IP 白名单**：内部服务只允许内网 IP 段访问

改进方案：在内部服务加一个 Filter，校验请求来源 IP 或内部签名，非内部调用直接拒绝。

### Q3：Feign 调用链路上 userId 是怎么传递的？用了 TransmittableThreadLocal，普通 ThreadLocal 为什么不行？TTL 的原理是什么？如果用 WebClient 响应式调用，TTL 还能工作吗？

**答：**

传递链路：Gateway 从 Sa-Token 读 userId → 写入请求 Header → 下游服务 Filter 从 Header 取 userId → 存入 TransmittableThreadLocal → 业务代码通过 `LoginUserContextHolder.getUserId()` 获取 → Feign 拦截器从 TTL 取 userId → 写入下游请求 Header。

**普通 ThreadLocal 不行的原因**：当使用线程池（@Async、CompletableFuture 等）时，子线程无法继承父线程的 ThreadLocal 值，因为 ThreadLocal 是和当前线程绑定的，线程池中的线程是复用的，不会每次都创建新线程。

**TTL 原理**：
- TTL 继承自 ThreadLocal，内部维护了一个 `holder`（WeakHashMap）记录所有 TTL 实例
- 在 `set()` 时会注册到当前线程的 holder 中
- 使用 `TtlRunnable.get()` 包装 Runnable，在任务执行前捕获父线程所有 TTL 值，执行时恢复到子线程，执行后还原
- 本质是**抓取-恢复-还原**三步操作

**WebClient 响应式调用**：TTL **不能直接工作**，因为响应式编程基于事件循环（Netty EventLoop），线程模型完全不同。需要用 Reactor Context 代替 ThreadLocal，在 `Mono.deferContextual()` 中传递 userId。

### Q4：Feign 调用链路上，Feign 底层走的什么 HTTP 客户端？有连接池配置吗？超时怎么设的？如果下游服务挂了，Feign 有重试机制吗？重试和 Ribbon 负载均衡的重试会不会叠加导致请求放大？

**答：**

- **底层 HTTP 客户端**：默认是 `HttpURLConnection`（JDK 原生，无连接池），生产环境通常替换为 **Apache HttpClient** 或 **OkHttp**，两者都支持连接池
- **连接池配置**：通过 `feign.httpclient.max-connections`、`feign.httpclient.max-connections-per-route` 配置
- **超时配置**：`feign.client.config.default.connectTimeout`（连接超时，通常 3-5s）、`feign.client.config.default.readTimeout`（读取超时，通常 5-10s）
- **重试机制**：Feign 默认不重试（`Retryer.NEVER_RETRY`），但 Ribbon 层有重试配置：`ribbon.MaxAutoRetriesNextServer`（换实例重试次数）和 `ribbon.MaxAutoRetries`（同实例重试次数）

**重试叠加问题**：如果 Feign 重试 + Ribbon 重试同时开启，最坏情况下请求次数 = `(1 + MaxAutoRetries) × (1 + MaxAutoRetriesNextServer)`，比如各配1次，最坏4次请求。**建议关闭 Feign 层重试，只保留 Ribbon 层重试**，且非幂等接口（POST）不重试。

### Q5：Nacos 既做注册中心又做配置中心，Nacos 的 AP/CP 模式怎么选的？临时实例和持久实例的区别？Nacos 集群间数据怎么同步的？如果 Nacos 挂了，服务还能调用吗？

**答：**

- **AP/CP 模式**：Nacos 默认临时实例用 AP 模式（Distro 协议，类似 Gossip），持久实例用 CP 模式（Raft 协议）。本项目用临时实例（AP），因为微服务实例经常上下线，可用性优先于一致性
- **临时实例 vs 持久实例**：
  - 临时实例：客户端主动心跳注册，Nacos 不健康则自动剔除，适合微服务
  - 持久实例：服务端主动探测健康状态，不健康标记但不剔除，适合数据库等基础设施
- **集群间同步**：AP 模式用 Distro 协议（每个节点负责一部分数据，异步广播给其他节点）；CP 模式用 Raft 协议（Leader 写入 + Follower 同步）
- **Nacos 挂了**：客户端有本地缓存（`nacos/naming` 目录），服务发现仍可工作，但新注册的服务无法被发现，配置也无法刷新。**建议 Nacos 集群部署至少3节点**

## 二、缓存篇

### Q6：你用了三级缓存 Caffeine → Redis → MySQL，本地缓存和分布式缓存的一致性怎么保证？你用 RocketMQ 广播通知各节点失效，如果 MQ 消息丢了怎么办？如果先删缓存再更新DB，在高并发下会出现什么问题？

**答：**

**一致性保证**：采用"延迟双删 + MQ广播失效"策略：
1. 先删除 Redis 缓存
2. 更新 DB
3. 发送延迟 MQ 消息（1s后再次删除 Redis）
4. 发送广播 MQ 消息（通知所有节点删除 Caffeine 本地缓存）

**MQ 消息丢了怎么办**：
- RocketMQ 支持消息追踪（trace），可以排查丢失
- 消费端有重试机制（默认16次）
- 极端情况下，本地缓存有过期时间（1h TTL），最终会自动失效
- 可以加一个兜底：数据对齐任务定期刷新热点缓存

**先删缓存再更新DB的并发问题**：

```
线程A: 删除缓存 → (还没更新DB)
线程B: 读缓存miss → 读DB(旧值) → 写缓存(旧值)
线程A: → 更新DB(新值)
结果: 缓存是旧值，DB是新值，数据不一致！
```

这就是延迟双删的意义——第二次删除会清掉线程B写入的旧缓存。

---

### Q7：你提到了"延迟双删"策略，为什么要延迟1秒？这个时间怎么确定的？如果第二次删除失败了呢？延迟双删能保证强一致性还是最终一致性？

**答：**

- **延迟1秒的原因**：这个时间要大于"读DB + 写缓存"的最大耗时。如果读DB+写缓存最多200ms，那延迟1秒足够覆盖。如果设太短，线程B还没来得及写旧缓存，第二次删除就执行了，删了个寂寞
- **确定方法**：理论上应该基于读DB+写缓存的 P99 耗时来设，加一定安全余量。也可以用 MQ 延迟消息实现，避免 Thread.sleep 占用线程
- **第二次删除失败**：缓存中残留旧数据，直到缓存自然过期（Redis TTL 1天+随机），或下次更新时再次触发双删。是**最终一致性**，不是强一致性
- **一致性级别**：延迟双删是**最终一致性**。要实现强一致性，需要加分布式读写锁（读加共享锁，写加排他锁），但性能代价很大

### Q8：缓存穿透你用缓存空值解决，如果大量不同 key 同时查询不存在的数据，空值缓存会占大量内存，怎么优化？布隆过滤器能解决吗？布隆过滤器有什么缺陷？

**答：**

- **空值缓存的问题**：恶意攻击构造大量随机不存在的 key，每个都缓存空值，Redis 内存会被打满
- **优化方案**：
  1. **布隆过滤器前置拦截**：在查缓存前先查布隆过滤器，不存在直接返回，不查DB也不缓存空值
  2. **空值缓存加短TTL + 限流**：空值TTL设极短（如10s），同时接口加限流
  3. **布隆过滤器 + 空值缓存结合**：布隆过滤器拦截大部分，漏网鱼（误判通过的）用空值缓存兜底

- **布隆过滤器缺陷**：
  1. **有误判率**：不存在的 key 可能被判为存在（false positive），但存在的 key 不会判为不存在（no false negative）
  2. **不支持删除**：多个 key 可能映射到同一个 bit 位，删除一个会影响其他 key
  3. **容量固定**：创建时需预估数据量，超出后误判率急剧上升
  4. **不支持计数查询**

  可用 **Cuckoo Filter**（布谷鸟过滤器）替代，支持删除操作。

---

### Q9：缓存雪崩你用随机 TTL 解决，如果是热点 key 过期导致大量请求打到 DB 呢？除了随机 TTL，还有什么方案？

**答：**

- **随机 TTL**：防止大量 key 同时过期，TTL = base + random(0, 300s)，打散过期时间
- **热点 key 过期方案**：
  1. **互斥锁（Mutex Lock）**：只让一个线程去查DB并回写缓存，其他线程等待。用 Redis `SET key value NX EX 30` 实现分布式锁
  2. **永不过期 + 异步刷新**：逻辑上永不过期（TTL 不设或设很长），后台定时任务或 MQ 触发异步刷新缓存数据
  3. **多级缓存**：本地缓存（Caffeine）作为兜底，Redis 过期后还有本地缓存挡一层
  4. **热点探测 + 本地预热**：运行时识别热点 key（如通过访问计数），提前将热点数据加载到本地缓存

本项目用的是**方案3（多级缓存）+ 随机TTL**的组合策略。

### Q10：Redis 你用了 String、Hash、ZSET、Bitmap、Bloom Filter 等多种数据结构，ZSET 底层怎么实现的？跳表的查询复杂度？为什么不用红黑树？Roaring Bitmap 和普通 Bitmap 的区别？

**答：**

- **ZSET 底层**：元素少时用 `ziplist`（压缩列表，省内存），元素多时用 `skiplist + dict`（跳表做范围查询，字典做 O(1) 按成员查分数）
- **跳表查询复杂度**：O(log N)，通过多层索引实现，每层约以 1/4 概率提升
- **为什么不用红黑树**：
  1. 跳表实现更简单，代码量少，不容易出 bug
  2. 跳表范围查询更高效（找到起点后沿底层链表遍历即可），红黑树需要中序遍历
  3. 跳表插入/删除只需修改局部指针，红黑树可能需要多次旋转
  4. 跳表的空间复杂度 O(N) 和红黑树相同，但常数因子略大

- **Roaring Bitmap vs 普通 Bitmap**：
  - 普通 Bitmap：每个整数占 1 bit，不管值是否存在。存 1 和 1000000 需要约 125KB 内存
  - Roaring Bitmap：将 32 位整数按高 16 位分桶（Container），每个 Container 65536 个值：
    - 稀疏容器（< 4096 个值）用 **Array Container**（short 数组，2 bytes/值）
    - 稠密容器（≥ 4096 个值）用 **Bitmap Container**（8KB 固定）
    - 连续值用 **Run Container**（行程编码）
  - **优势**：稀疏数据集下内存远小于普通 Bitmap；支持快速交集/并集运算

## 三、消息队列篇

### Q11：发布笔记用了 RocketMQ 事务消息，事务消息的完整流程是什么？half消息、回查机制怎么工作的？如果本地事务一直不提交，MQ 会怎么处理？事务消息和本地消息表方案比，各有什么优缺点？

**答：**

**事务消息完整流程**：
1. Producer 发送 half 消息（对 Consumer 不可见）到 Broker
2. Broker 存储半消息到内部特殊 Topic（`RMQ_SYS_TRANS_HALF_TOPIC`），返回发送成功
3. Producer 执行本地事务（如插入笔记元数据到 MySQL）
4. 根据本地事务结果，Producer 发送 commit/rollback 给 Broker
5. commit → Broker 将消息移到目标 Topic，Consumer 可见；rollback → Broker 删除消息

**回查机制**：
- 如果 Broker 长时间未收到 commit/rollback（默认 15s），会启动回查线程
- 回查 Producer 的 `checkLocalTransaction()` 方法，Producer 根据本地事务状态返回 commit/rollback/unknown
- 最多回查 15 次，间隔递增（10s, 30s, 1min...），超过则 rollback

**本地事务一直不提交**：回查不断触发，最终超过最大回查次数后消息被 rollback 丢弃。

**事务消息 vs 本地消息表**：

| 维度 | 事务消息 | 本地消息表 |
|------|---------|-----------|
| 一致性保证 | MQ 侧保证 | DB 侧保证 |
| 额外存储 | 无 | 需要额外表 |
| 实现复杂度 | 中（需写回查逻辑） | 高（需定时扫描+重发） |
| MQ 依赖 | 强依赖 RocketMQ | 任何 MQ 都行 |
| 性能 | 高（无额外 DB 写入） | 略低（每次多写一条消息记录） |

### Q12：点赞/收藏用了顺序消息（hashKey = userId），顺序消息的原理？同一个 Queue 里消息有序，但 Consumer 多线程消费怎么办？你们怎么保证消费顺序的？如果某个 userId 的消息特别多，会不会造成数据倾斜？

**答：**

**顺序消息原理**：
- Producer 端：相同 hashKey（userId）的消息会被 hash 到同一个 Queue
- Broker 端：同一个 Queue 内的消息严格按存储顺序投递
- Consumer 端：同一个 Queue 同一时刻只有一个线程消费

**保证消费顺序**：
- RocketMQ 的 `MessageListenerOrderly` 保证同一 Queue 单线程顺序消费
- 如果用 `MessageListenerConcurrently`，同一 Queue 可能多线程并发消费，顺序无法保证
- 本项目使用 `MessageListenerOrderly`，同一 userId 的点赞/取消点赞严格有序

**数据倾斜问题**：
- 如果某个超级大 V userId 的消息特别多，其所在 Queue 会成为热点
- 解决方案：
  1. **合理设置 Queue 数量**：Queue 数量 = Consumer 实例数 × 每实例消费线程数
  2. **hashKey 选择**：如果 userId 分布不均，可以加随机前缀打散（但会破坏同一用户的顺序性，需要权衡）
  3. **本项目中**：userId 天然分散，大 V 的操作频率虽高但不会集中到极端程度

### Q13：MQ 消费者必须幂等，你项目里怎么实现幂等的？唯一索引？状态机？Redis 去重？如果幂等判断和业务操作不是原子的怎么办？

**答：**

**项目中的幂等实现**：
1. **唯一索引**：如 `t_note_like` 表的 (userId, noteId) 联合唯一索引，重复插入会抛 `DuplicateKeyException`，捕获后忽略
2. **业务状态判断**：消费前先查 Redis（如 Roaring Bitmap 判断是否已点赞），已处理则直接返回
3. **Redis 去重**：用 `SET messageId NX EX 3600` 去重，消息消费前先设 key，设成功才消费

**幂等判断和业务操作非原子的问题**：
- 场景：先查 Redis 未处理 → 还没执行业务 → 另一个线程也查到未处理 → 重复执行
- 解决方案：
  1. **Redis Lua 脚本**：将去重判断和业务操作放在同一个 Lua 脚本中原子执行
  2. **唯一索引兜底**：即使 Redis 层没拦住，DB 唯一索引是最终防线
  3. **先标记后执行**：先 `SETNX` 标记为"处理中"，再执行业务，成功后更新为"已完成"

本项目主要依赖 **Redis 快速判断 + DB 唯一索引兜底** 的双层幂等保障。

### Q14：MQ 发送失败怎么办？你用了异步发送 + SendCallback，onException 里只是打了日志，消息不就丢了吗？有没有本地重试？有没有落库重发？Comment 服务有 SendMqRetryHelper，具体怎么实现的？

**答：**

**当前问题**：异步发送 `onException` 只打日志，消息确实可能丢失。这在生产环境中是不可接受的。

**SendMqRetryHelper 实现思路**：
- 封装发送逻辑，内置本地重试（如重试3次，间隔递增）
- 重试仍失败则：
  1. **落本地表**：将消息内容写入本地消息表（状态=待发送）
  2. **定时扫描重发**：后台定时任务扫描待发送消息，重新发送
  3. **发送成功后标记**：更新消息状态为已发送
  4. **超过最大重试次数告警**：人工介入处理

**更完善的方案**：
1. **同步发送**：关键业务（如事务消息）用同步发送，失败直接抛异常回滚
2. **本地消息表 + 定时补偿**：所有异步消息先落本地表，发送成功后标记，定时扫描补偿
3. **RocketMQ 内置重试**：Producer 有 `retryTimesWhenSendFailed`（同步重试次数，默认2）和 `retryTimesWhenSendAsyncFailed`（异步重试次数，默认2）

## 四、数据库篇

### Q15：你用 MySQL 存元数据、Cassandra 存长文本，为什么要这样拆？MySQL 存长文本有什么问题？Cassandra 的写入模型是什么？和 B+ Tree 比各适合什么场景？

**答：**

**MySQL 存长文本的问题**：
1. **页分裂**：InnoDB 默认 16KB 页，长文本（如笔记内容 5000 字）可能跨多页存储，INSERT/UPDATE 导致频繁页分裂，IO 放大
2. **Buffer Pool 污染**：大字段占满 Buffer Pool，把热点小数据挤出，降低缓存命中率
3. **查询效率低**：`SELECT *` 会把大字段一起加载，即使只需要元数据
4. **索引效率下降**：B+ Tree 节点变大，树变矮胖，单次 IO 加载的记录数减少

**Cassandra 写入模型**：
- 基于 **LSM Tree**（Log-Structured Merge Tree）
- 写入流程：先写 CommitLog（顺序写，持久化保证）→ 写 MemTable（内存排序树）→ MemTable 满后刷盘为 SSTable（不可变，顺序写）→ 后台 Compaction 合并 SSTable
- **优势**：写入是顺序写，无随机 IO，写入吞吐远超 B+ Tree
- **劣势**：读取可能需要查多个 SSTable（除非在 MemTable 或 Bloom Filter 命中缓存），读放大

**适用场景**：
- **B+ Tree（MySQL）**：读多写少、需要事务、复杂查询、范围扫描
- **LSM Tree（Cassandra）**：写多读少、超大规模数据、高可用容忍最终一致、无需复杂查询

本项目笔记内容写一次读多次，但内容体量大、无需事务，Cassandra 适合存储。

### Q16：数据对齐服务用 XXL-JOB 分片广播，每天建临时表做增量对齐，为什么要用临时表而不是直接查源表？分片广播的原理？如果某片执行失败怎么办？对齐过程中如果又有新的点赞数据写入，会不会漏算？

**答：**

**用临时表的原因**：
1. **避免锁表**：直接在源表上做 COUNT + UPDATE 会持有锁，影响在线业务
2. **增量计算**：临时表只存当天新增的记录（如 `t_data_align_note_like_count_temp_20240101_0`），计算量小
3. **并行处理**：分片后每个分片处理自己的临时表，互不干扰
4. **可追溯**：临时表保留对齐过程数据，出问题可排查
5. **物理删除已处理记录**：对齐完成后删除临时表数据，释放空间

**分片广播原理**：
- XXL-JOB 调度中心向所有执行器实例发送同一任务
- 每个实例获取自己的分片参数：`shardIndex`（分片序号）和 `shardTotal`（总分片数）
- 业务代码按 `id % shardTotal == shardIndex` 分配数据，每个实例只处理自己分片的数据

**某片执行失败**：
- XXL-JOB 会记录执行日志，该片数据未对齐
- 第二天对齐任务会重新处理（临时表中未处理的数据还在）
- 也可以手动触发重跑

**新写入数据漏算问题**：
- 临时表是当天增量快照，对齐期间新写入的数据不在临时表中
- 但这些新数据会在**第二天的对齐任务**中被覆盖
- 计数服务本身是 Redis 实时计数，DB 计数只是对齐后的结果，用户看到的始终是 Redis 的值

### Q17：Druid 连接池密码加密，Druid 的 RSA 加密原理？公钥放在配置文件里安全吗？有没有更安全的方案？连接池的核心参数怎么调优的？

**答：**

**Druid RSA 加密原理**：
1. 用 `ConfigTools.encrypt(password)` 生成密文（RSA 公钥加密）
2. 配置文件中写 `password: 密文` + `connectionProperties: config.decrypt=true;config.decrypt.key=公钥`
3. Druid 初始化时用公钥解密密文得到明文密码

**公钥放配置文件的安全问题**：
- 公钥本身不是秘密，RSA 安全性依赖私钥保密
- 但如果攻击者拿到公钥 + 密文，虽然无法解密，但可以用私钥重新加密一个已知密码替换
- **更安全的方案**：
  1. **Vault/HashiCorp Vault**：密码存在 Vault 中，应用启动时从 Vault 拉取
  2. **KMS（密钥管理服务）**：阿里云/AWS KMS，密码加密存储，运行时解密
  3. **K8s Secret**：密码存 K8s Secret，以环境变量注入
  4. **Nacos 加密配置**：Nacos 2.x 支持配置加密

**连接池核心参数调优**：
- `maxActive`（最大活跃连接数）：= (DB CPU 核数 * 2) + 有效磁盘数，或按公式 `connections = ((core_count * 2) + effective_spindle_count)`
- `minIdle`（最小空闲连接）：= maxActive，避免连接创建开销
- `maxWait`（获取连接最大等待时间）：3000-5000ms，超时报错
- `timeBetweenEvictionRunsMillis`（检测间隔）：60000ms
- `minEvictableIdleTimeMillis`（最小空闲时间）：300000ms
- 核心原则：**连接数不是越多越好**，过多会导致 DB 端上下文切换开销增大、锁争用加剧

### Q18：你用了 38 个 Lua 脚本，为什么不用 pipeline？Lua 脚本执行时 Redis 会阻塞吗？如果 Lua 脚本执行时间过长会怎样？Redis 7.0 的 Function 和 Eval 比有什么改进？

**答：**

**Lua vs Pipeline**：
- **Pipeline**：只是批量发送命令减少网络 RTT，但多个命令之间**不是原子的**，其他客户端的命令可能穿插执行
- **Lua 脚本**：整个脚本作为**一个原子操作**执行，中间不会被其他命令打断，适合需要"判断+操作"原子的场景（如：判断是否已点赞 + 添加点赞记录 + 更新计数）

**Redis 执行 Lua 的阻塞问题**：
- Redis 是单线程模型，执行 Lua 脚本期间**整个 Redis 被阻塞**，无法处理其他请求
- 如果 Lua 执行时间超过 `lua-time-limit`（默认 5000ms），Redis 会开始接受并处理其他客户端的 `SCRIPT KILL` 或 `SHUTDOWN NOSAVE` 命令
- 但如果 Lua 正在执行写操作，`SCRIPT KILL` 无法终止，只能 `SHUTDOWN NOSAVE`

**最佳实践**：
- Lua 脚本必须**短小快速**，不能有循环遍历大集合
- 不能在 Lua 中做耗时操作（如大范围 ZRANGE）
- 本项目的 Lua 脚本都很短（判断存在 + 更新 ZSET + 更新计数），执行在微秒级

**Redis 7.0 Function vs Eval**：
- **Eval**：每次发送脚本全文，有解析开销；脚本不持久化，重启丢失
- **Function**：预先注册到 Redis（类似存储过程），持久化到 RDB/AOF；执行时只需调用函数名，无解析开销；支持 Function 库管理（LIST、DELETE、FLUSH）
- 本项目仍用 Eval，因为脚本数量有限且变更灵活

## 五、分布式 ID 篇

### Q19：你同时用了 Leaf Segment 和 Snowflake，两种方案各适合什么场景？Segment 方案 DB 挂了怎么办？Snowflake 的 workerId 怎么分配的？时钟回拨问题怎么解决？

**答：**

**适用场景**：
- **Leaf Segment**：适合**趋势递增**且**不依赖时间**的场景，如用户 ID、笔记 ID，对 DB 友好（有序写入，B+ Tree 不分裂）
- **Snowflake**：适合**高并发**且**需要分布式**的场景，如评论 ID、消息 ID，不依赖 DB，性能更高

**Segment 方案 DB 挂了怎么办**：
- **双 Buffer 机制**：每个业务 key 维护两个 Segment（Segment1 和 Segment2）
- 当前用 Segment1 发号，当 Segment1 用到 10% 时，异步加载 Segment2
- Segment1 用完后切换到 Segment2，同时异步加载新的 Segment1
- DB 短暂不可用时，Buffer 中还有未用完的号段，可以继续发号
- 但如果 DB 长时间不可用且 Buffer 耗尽，就无法发号了

**Snowflake workerId 分配**：
- 本项目用 ZooKeeper 分配：每个实例启动时在 ZK 的持久顺序节点下创建临时顺序节点，节点序号即为 workerId
- 实例下线后临时节点自动删除，workerId 可复用

**时钟回拨解决**：
1. **检测回拨**：每次获取时间戳时和上次比较，如果当前时间 < 上次时间，说明回拨了
2. **轻度回拨**（< 5ms）：Thread.sleep 等待追上
3. **中度回拨**（< 500ms）：使用历史最大时间戳 + 1
4. **严重回拨**（> 500ms）：直接抛异常，拒绝生成 ID
5. **Leaf 的方案**：用 ZK 保存历史最大时间戳，回拨时和 ZK 时间比较

### Q20：Snowflake ID 是趋势递增的，这对 MySQL 索引有什么好处？但如果用 Snowflake ID 做 Redis ZSET 的 score，会不会精度丢失？你们怎么处理的？

**答：**

**对 MySQL 索引的好处**：
- 趋势递增意味着新数据总是追加到 B+ Tree 的右侧，不会引起页分裂
- 索引树保持紧凑，空间利用率高
- 顺序写入性能远优于随机写入（UUID 的问题）

**Redis ZSET 精度丢失问题**：
- Redis ZSET 的 score 是 **64 位双精度浮点数（double）**
- double 的有效位数是 **53 位**（尾数部分）
- Snowflake ID 是 64 位整数，当 ID > 2^53（约 9007 万亿）时，精度丢失
- 实际上 Snowflake 的 41 位时间戳 + 10 位 workerId + 12 位序列号 = 63 位，最大值约 2^63，远超 2^53

**本项目处理方式**：
- 关注列表、粉丝列表用 ZSET 存储，score 是时间戳（毫秒级，41位），在 double 精度范围内，不会丢失
- 如果需要存 Snowflake ID 作为 score，可以：
  1. 用 **Redis 7.0+ 的 Sorted Set 原生整数 score**（如果可用）
  2. 将 ID 转为字符串存 ZSET，用自定义排序
  3. 用 **Hash + 手动排序** 代替 ZSET

## 六、搜索与推荐篇

### Q21：你用 Canal 监听 binlog 同步 ES，Canal 的原理？如果 Canal 宕机一段时间，重启后怎么追上进度？如果同步顺序错了怎么办？ES 写入是原子的吗？

**答：**

**Canal 原理**：
1. Canal 伪装成 MySQL 的从节点（Slave），向 Master 发送 `COM_BINLOG_DUMP` 请求
2. MySQL Master 接收到 dump 请求后，开始推送 binlog 事件给 Canal
3. Canal 解析 binlog 事件（INSERT/UPDATE/DELETE），提取变更数据
4. 将变更数据发送到下游（本项目是 ES）

**Canal 宕机重启追进度**：
- Canal 持久化消费位点（binlog filename + position）到 ZooKeeper 或本地文件
- 重启后从上次记录的位点继续消费，自动追上
- 如果位点丢失，可以手动指定起始位点

**同步顺序问题**：
- binlog 本身是按事务提交顺序记录的，Canal 按顺序消费，单线程同步到 ES 时顺序正确
- 如果多线程/多 Consumer 并发同步 ES，可能乱序
- 解决：单线程同步，或用版本号/时间戳做乐观锁

**ES 写入原子性**：
- ES 单文档的写入是原子的（单文档级别 ACID）
- 但跨文档/跨索引的操作不是原子的
- ES 的写入流程：Write → Refresh（1s 默认）→ 可搜索。刚写入的数据可能 1s 内搜不到
- 可以用 `refresh=true` 强制立即可搜索，但性能差

### Q22：推荐服务用了三路召回，为什么用三路而不是更多？召回结果怎么合并去重的？ES function_score 里的公式为什么用 sqrt 而不是原始值？Gauss 衰减的 half_life 设 7 天怎么确定的？

**答：**

**三路召回的原因**：
- **热门召回**：保证新用户有内容可看（冷启动）
- **关注召回**：保证社交属性，用户看到关注的人的内容
- **话题召回**：基于兴趣偏好，个性化推荐
- 三路覆盖了"大众兴趣 + 社交关系 + 个人偏好"三个维度，更多路召回（如协同过滤、内容相似度）增加计算成本但边际收益递减

**合并去重**：
- 用 `Set<Long>` 按 noteId 去重
- 优先级：关注 > 话题 > 热门（关注的内容排在前面）
- 去重后按优先级和分数综合排序，截取 Top N

**为什么用 sqrt**：
- 点赞数、收藏数、评论数是**长尾分布**（少数爆款极高，大多数很低）
- 如果用原始值，爆款分数远超普通笔记，普通笔记永远排不上
- sqrt 做了**非线性压缩**：10000 → 100，100 → 10，差距从 100 倍缩小到 10 倍
- 也可以用 log 压缩，sqrt 压缩力度介于线性和 log 之间

**half_life 7 天怎么确定**：
- 小红书内容生命周期约 1-2 周，7 天半衰期意味着 7 天前的内容权重降为 50%，14 天降为 25%
- 可以根据业务数据分析内容热度衰减曲线来调整
- 如果是新闻类应用，half_life 可能设 1-2 天；如果是知识类应用，可能设 30 天

### Q23：ES 搜索建议怎么实现的？completion suggester 和 term suggester 的区别？你们用的是哪种？ES 倒排索引的原理？FST 数据结构了解吗？

**答：**

**搜索建议实现**：本项目用 ES 的 **Completion Suggester**，在 note 索引中定义 `title.suggest` 字段（type: completion），写入时自动构建 FST 前缀索引。

**Completion vs Term Suggester**：

| 维度 | Completion Suggester | Term Suggester |
|------|---------------------|----------------|
| 原理 | 前缀匹配（FST） | 编辑距离（Levenshtein） |
| 用途 | 搜索补全/自动补全 | 拼写纠错/模糊建议 |
| 性能 | 极快（O(1) 前缀查找） | 较慢（需计算编辑距离） |
| 数据结构 | FST（有限状态转换器） | 倒排索引 + 编辑距离 |
| 索引时开销 | 需要额外字段，索引时构建 | 无额外开销 |

**倒排索引原理**：
1. **分词**：对文档内容分词，得到 Token 流
2. **建立倒排表**：Token → [DocId1, DocId2, ...]，记录每个词出现在哪些文档中
3. **记录词频和位置**：每个 (Token, DocId) 对还记录词频 TF、位置 Position、偏移 Offset
4. **查询时**：对查询词分词 → 查倒排表 → 交集/并集 → 打分排序 → 返回

**FST（Finite State Transducer）**：
- 是一种**有限状态机**，同时编码了 key → value 的映射
- 比 Trie 树更省内存：共享前缀 + 共享后缀
- 查询复杂度 O(key_length)
- Completion Suggester 用 FST 存所有分词的前缀，实现毫秒级前缀补全
- Lucene 中 FST 还用于 Term Dictionary 的查找（替代二分查找）

## 七、AI Agent 篇

### Q24：你用 Spring AI + 智谱 GLM-4 做多 Agent 编排，Agent 和普通 LLM 调用的区别？Tool Calling 的原理？LLM 怎么知道什么时候该调 Tool？Function Call 的协议格式是什么？

**答：**

**Agent vs 普通 LLM 调用**：
- **普通 LLM 调用**：单轮问答，Prompt → LLM → Response，无外部交互
- **Agent**：LLM + Tool + Memory + Planning，LLM 可以自主决定调用哪些 Tool、调用顺序、根据结果调整策略，形成 **"思考-行动-观察"循环（ReAct 模式）**

**Tool Calling 原理**：
1. 在请求中声明可用 Tools（名称、描述、参数 JSON Schema）
2. LLM 根据用户输入和 Tool 描述，决定是否需要调用 Tool
3. 如果需要，返回 `tool_calls`（Tool 名称 + 参数），而不是普通文本
4. 应用层执行 Tool，将结果作为 `tool` 角色消息追加到对话
5. 再次调用 LLM，LLM 根据工具结果生成最终回复（或继续调用其他 Tool）

**LLM 怎么知道何时调 Tool**：
- 依赖 Tool 的 **description** 描述，LLM 在训练时学过 Function Call 的模式
- System Prompt 也可以引导 LLM 何时使用 Tool
- 本质是 LLM 的**指令遵循能力**，不是硬编码逻辑

**Function Call 协议格式**（OpenAI 兼容）：
```json
{
  "tools": [{
    "type": "function",
    "function": {
      "name": "searchNotes",
      "description": "搜索笔记",
      "parameters": {
        "type": "object",
        "properties": {"keyword": {"type": "string"}},
        "required": ["keyword"]
      }
    }
  }],
  "messages": [{"role": "user", "content": "帮我搜小红书运营技巧"}]
}
```
LLM 返回：
```json
{
  "choices": [{"message": {
    "tool_calls": [{"function": {"name": "searchNotes", "arguments": "{\"keyword\": \"小红书运营技巧\"}"}}]
  }}]
}
```

### Q25：四种编排模式（sequential/parallel/conditional/guarded），parallel 模式下两个 Agent 结果冲突怎么办？guarded 模式里 moderation 不通过是直接拒绝还是降级？Agent 之间怎么共享上下文的？

**答：**

**parallel 结果冲突**：
- 本项目中并行执行的是 ContentUnderstanding 和 ContentModeration，两者职责不同（理解内容 vs 审核内容），不会产生冲突
- 如果两个 Agent 对同一问题给出矛盾结论，需要设计**冲突解决策略**：
  1. 优先级策略：指定一个 Agent 的结果优先
  2. 投票策略：多 Agent 投票，少数服从多数
  3. 裁决策略：引入第三个 Agent 做裁判

**moderation 不通过的处理**：
- guarded 模式下，moderation 不通过时**直接拒绝**，不进入 CreativeAssistant
- 返回审核不通过的原因给用户
- 也可以设计**降级策略**：moderation 不通过时，CreativeAssistant 只能在安全范围内回答（如只推荐已有内容，不生成新内容）

**Agent 间共享上下文**：
- 通过 **ChatMemory**（Redis 实现）共享对话历史
- 前一个 Agent 的输出作为后一个 Agent 输入的一部分
- 具体实现：`AgentOrchestrator` 将前序 Agent 的结果追加到 `ChatMessageList`，传给下一个 Agent
- 所有 Agent 共享同一个 `conversationId` 对应的 Redis 记录

### Q26：你对 Tool 调用加了 Sentinel 限流，为什么？LLM 会不会在一次对话中疯狂调 Tool？限流后返回 fallback 结果，LLM 拿到 fallback 会怎么处理？有没有考虑过在 System Prompt 里限制 Tool 调用次数？

**答：**

**为什么要限流**：
- LLM 可能陷入 **Agent Loop**：调 Tool → 结果不满意 → 再调 Tool → 循环不止
- Tool 调用会触发 ES 搜索等 IO 操作，无限循环会压垮底层服务
- 多用户并发时，Tool 调用数 = 用户数 × 每用户调用次数，可能瞬间打满 ES

**LLM 疯狂调 Tool 的场景**：
- 搜索关键词太宽泛，结果太多，LLM 反复细化搜索
- Tool 返回结果格式不符合 LLM 预期，LLM 认为调用失败，反复重试
- LLM 的推理出错，陷入死循环

**Fallback 结果 LLM 怎么处理**：
- 本项目的 `ToolFallbackHandler` 返回 "服务繁忙，请稍后再试" 之类的提示
- LLM 会将此作为 Tool 返回结果，生成类似"抱歉，搜索服务暂时不可用"的回复
- 但 LLM 可能**再次尝试调用 Tool**，需要配合最大 Tool 调用次数限制

**System Prompt 限制调用次数**：
- 可以在 System Prompt 中加入 "你最多只能调用3次搜索工具"
- 但 LLM 不保证 100% 遵循指令，**硬限制更可靠**
- Spring AI 支持 `maxToolCalls` 配置，超过后不再允许 Tool 调用，强制 LLM 基于已有信息回答

## 八、安全与高可用篇

### Q27：Sa-Token 基于 Redis 存 Session，和 JWT 比各有什么优缺点？Sa-Token 能主动踢人下线，JWT 能吗？Redis 挂了 token 全失效怎么办？

**答：**

| 维度 | Sa-Token (Redis Session) | JWT |
|------|-------------------------|-----|
| 状态 | 有状态（Redis 存储） | 无状态（自包含） |
| 主动踢人 | ✅ 删 Redis key 即可 | ❌ 无法主动失效（除非加黑名单，又变成有状态） |
| 续期 | ✅ 调 API 续期 | ❌ 需要刷新令牌机制 |
| 性能 | 每次鉴权需查 Redis | 自解密，无需远程调用 |
| 水平扩展 | 依赖 Redis 集群 | 天然支持 |
| 信息量 | token 本身无业务信息 | payload 可携带用户信息 |
| 安全性 | token 泄露可立即失效 | token 泄露在过期前无法失效 |

**Redis 挂了 token 全失效**：
- 所有用户被强制下线，需要重新登录
- 应对方案：
  1. **Redis 高可用**：Sentinel 哨兵模式或 Cluster 集群，主从自动故障转移
  2. **Redis 持久化**：开启 AOF 或 RDB，重启后数据可恢复
  3. **降级方案**：Redis 不可用时切换为本地 Session（单节点可用，集群不行）
  4. **多级存储**：token 同时存 Redis + 本地缓存，Redis 挂了用本地缓存兜底（但踢人功能失效）

### Q28：RBAC 权限数据启动时推到 Redis，如果运行时修改了角色权限，Redis 里的数据怎么更新？有权限缓存失效机制吗？权限校验在 Gateway 层做的，如果内部服务被直接调用，权限校验还能生效吗？

**答：**

**运行时权限更新**：
- 当前项目 `PushRolePermissions2RedisRunner` 只在启动时推一次
- 运行时修改角色权限后，需要手动调用接口或发 MQ 消息触发 Redis 刷新
- 改进方案：
  1. **管理后台修改权限后发 MQ** → 所有 User 服务实例消费 → 重新加载权限到 Redis
  2. **Nacos 配置变更通知**：权限版本号放 Nacos，变更时推送通知
  3. **定时刷新**：每隔 N 分钟重新加载一次（简单但有延迟）

**内部服务被直接调用**：
- Gateway 层的 RBAC 校验**不生效**，内部服务没有独立的权限校验
- 这是当前架构的安全盲区
- 改进方案：
  1. **内部服务加 AOP 鉴权**：自定义注解 `@RequirePermission`，AOP 拦截校验
  2. **内部服务间信任**：内部服务间调用不需要权限校验，只对外部请求校验（Gateway 层已做）
  3. **Service Mesh**：Istio 等可以在网格层做权限校验，不依赖 Gateway

### Q29：Sentinel 限流用在 Count 服务和 Agent Tool 上，Sentinel 的限流算法？热点参数限流怎么配的？簇点链路是什么概念？如果用 Sentinel 做熔断，慢调用比例和异常比例两种策略怎么选？

**答：**

**Sentinel 限流算法**：
- **滑动窗口统计**：默认算法，将时间窗口细分为多个小窗口（如 1s 分为 2 个 500ms 子窗口），统计时滑动计算，比固定窗口更平滑
- **令牌桶**：`RateLimiterController`，匀速发放令牌，允许突发流量（桶中有积累的令牌），适合平滑限流
- **漏桶**：匀速处理请求，不允许突发，适合严格控制处理速率

**热点参数限流**：
- 对接口的某个参数值单独限流，如 `@SentinelResource` + `ParamFlowItem` 对某个 noteId 限流
- 配置示例：`paramFlowKey` 指定参数索引，`paramFlowVal` 指定阈值，`paramFlowItem` 指定特定参数值的阈值

**簇点链路**：
- Sentinel 将每个被保护的资源（方法、URL）视为一个节点
- 调用链路上的节点组成"簇点链路"，如 `Gateway → /note/detail → findNoteDetail → Redis.get → MySQL.select`
- 可以在 Sentinel Dashboard 上看到每个节点的 QPS、RT、异常数等指标

**熔断策略选择**：
- **慢调用比例**：适合下游服务变慢但不报错的场景（如 DB 慢查询、网络抖动），设置最大 RT 阈值，超过则记为慢调用
- **异常比例/异常数**：适合下游服务直接报错的场景（如服务宕机、抛异常），统计异常比例/数量
- **选择原则**：下游可能"慢但不挂"用慢调用比例；下游"挂了报异常"用异常比例；两者都可能则**同时配置两条规则**

## 九、性能优化篇

### Q31：笔记详情用了 CompletableFuture 并行调用 User RPC 和 KV RPC，这里的线程池怎么配的？用的是 ForkJoinPool 还是自定义线程池？如果两个 RPC 都很慢，allOf 的超时怎么控制？CompletableFuture 有什么常见陷阱？

**答：**

**线程池配置**：
- 如果不指定线程池，`CompletableFuture.supplyAsync()` 默认用 `ForkJoinPool.commonPool()`，线程数 = CPU 核数 - 1
- 生产环境**必须自定义线程池**，原因：
  1. commonPool 是全局共享的，其他 CompletableFuture/parallelStream 也用它，互相影响
  2. 线程数不可控，IO 密集型任务需要更多线程
  3. 无法监控线程池指标（活跃线程数、队列长度等）

**超时控制**：
- `CompletableFuture.allOf()` 本身**不支持超时**
- Java 9+ 有 `orTimeout()` 和 `completeOnTimeout()`
- Java 8 的方案：
  ```java
  CompletableFuture.allOf(f1, f2)
    .get(3, TimeUnit.SECONDS); // 阻塞等待，超时抛 TimeoutException
  ```
  或用 `anyOf` + 超时 CompletableFuture 竞争

**常见陷阱**：
1. **异常吞没**：`join()`/`get()` 不加 try-catch，异常被吞；应用 `exceptionally()` 或 `handle()` 处理
2. **阻塞等待**：在 Web 容器线程中 `get()` 阻塞，占用 Tomcat 线程，应该返回 CompletableFuture 让框架处理
3. **线程池用错**：用 commonPool 做 IO 密集任务，线程不够导致任务排队
4. **依赖链过深**：f1 → f2 → f3 → f4，任何一环异常整链失败
5. **非原子读取**：`f1.isDone() && f2.isDone()` 不是原子操作，可能 f1 完成后 f2 状态变了

### Q32：BufferTrigger 做批量聚合写入，原理是什么？和直接一条条写比，QPS 提升多少？如果 Buffer 里的数据还没刷盘服务就挂了怎么办？数据会丢吗？

**答：**

**BufferTrigger 原理**：
- 来自 `com.github.lkq:buffer-trigger` 库
- 核心机制：**时间窗口 + 数量阈值双触发**
- 数据先写入内存 Buffer，满足以下任一条件触发刷盘：
  1. Buffer 中数据量达到 `bufferSize` 阈值
  2. 距上次刷盘超过 `timeInterval` 时间
- 刷盘时将 Buffer 中所有数据一次性批量写入 DB

**QPS 提升**：
- 假设单条写入 DB 耗时 5ms，批量写入 100 条耗时 50ms
- 单条写：1s / 5ms = 200 QPS
- 批量写：100 条 / 50ms = 2000 QPS，提升 10 倍
- 实际提升取决于批量写入的效率，通常 **5-20 倍**

**服务挂了数据丢失问题**：
- **会丢数据**：Buffer 中的数据还在内存中，服务宕机后无法恢复
- 本项目用于计数更新（点赞数、收藏数等），丢失的计数可以通过**数据对齐服务**补偿
- 如果不能接受丢失：
  1. 先写 WAL（预写日志），刷盘时同时清理 WAL
  2. 用 MQ 做缓冲（MQ 有持久化保证），Consumer 批量消费写入
  3. 用 Redis 做缓冲（Redis 有 AOF 持久化），后台任务批量读 Redis 写 DB

### Q33：MapStruct 做对象转换，和 BeanUtils.copyProperties 比有什么优势？MapStruct 的性能比反射方式快多少？MapStruct 不支持的场景有哪些？

**答：**

**MapStruct vs BeanUtils**：

| 维度 | MapStruct | BeanUtils (Spring) |
|------|-----------|-------------------|
| 原理 | 编译期代码生成 | 运行时反射 |
| 性能 | ≈ 手写 set/get | 慢 50-100 倍 |
| 类型安全 | ✅ 编译期检查类型匹配 | ❌ 运行时才发现类型不匹配 |
| 深拷贝 | ✅ 支持嵌套对象映射 | ⚠️ 浅拷贝，嵌套对象是引用 |
| 自定义映射 | ✅ @Mapping 注解灵活 | ❌ 只能同名同类型复制 |
| 字段名不同 | ✅ @Mapping(source, target) | ❌ 不支持 |

**性能对比**：
- MapStruct 生成的代码等价于手写 `dto.setXxx(do.getXxx())`，纯方法调用，纳秒级
- BeanUtils 每次调用需要：获取 Class → 获取 Fields → 遍历 → 反射 get/set → 类型转换，微秒级
- **MapStruct 快 50-100 倍**，在高频调用场景差异明显

**MapStruct 不支持的场景**：
1. **运行时动态映射**：字段映射关系在编译期确定，无法运行时改变
2. **循环引用**：A → B → A 会导致无限递归
3. **Map/List 泛型擦除**：泛型信息在编译期擦除，复杂泛型映射可能需要自定义方法
4. **多源合并**：一个 DTO 由多个 DO 合并而来需要 `@Mapping` 从多个 source 指定

### Q34：Roaring Bitmap 做点赞判断，和 Bloom Filter 比各有什么优劣？Roaring Bitmap 能删除元素吗？Bloom Filter 能吗？如果用户量到亿级，Roaring Bitmap 的内存占用是多少？

**答：**

**Roaring Bitmap vs Bloom Filter**：

| 维度 | Roaring Bitmap | Bloom Filter |
|------|---------------|-------------|
| 误判 | ❌ 无误判（精确判断） | ⚠️ 有误判（false positive） |
| 删除 | ✅ 支持删除 | ❌ 不支持删除 |
| 内存 | 中等（取决于数据密度） | 小（固定大小） |
| 交集/并集 | ✅ 高效位运算 | ❌ 不支持 |
| 适用场景 | 需要精确判断 + 删除 + 集合运算 | 只需判断"可能存在" + 内存敏感 |

**Roaring Bitmap 内存估算（亿级）**：
- 假设 1 亿用户，点赞用户 1000 万（10% 密度）
- Roaring Bitmap 将 32 位整数按高 16 位分桶（65536 个 Container）
- 1000 万个值分布在约 65536 个 Container 中，平均每个 Container 约 153 个值
- 稀疏 Container（< 4096）用 Array Container：153 × 2 bytes = 306 bytes/Container
- 总内存 ≈ 65536 × 306 bytes ≈ **20 MB**
- 对比普通 Bitmap：1 亿 / 8 = **12.5 MB**（固定，不管实际有多少值）
- 对比 Bloom Filter（1% 误判率）：约 **11.5 MB**

**结论**：亿级场景下三者内存差距不大，Roaring Bitmap 在需要精确判断和删除的场景下是最佳选择。

### Q35：如果让你给这个系统加一个"热搜榜"功能，你会怎么设计？

**答：**

**方案设计**：

1. **数据采集层**：
   - 搜索服务已有 ES，每次搜索词记录到 Redis ZSET（key: `hot_search:realtime`，score: +1）
   - 笔记发布/点赞/收藏事件通过 MQ 发送到热搜计算服务

2. **热度计算层**：
   - **实时热度**：Redis ZSET + Lua 原子加分，score = 搜索次数 × 1 + 点赞数 × 2 + 收藏数 × 3
   - **时间衰减**：定时任务每分钟对 ZSET 所有 score 乘以衰减因子（如 0.99），让旧热搜自然冷却
   - **去重**：同一用户对同一关键词的搜索，短时间内只计一次（Redis SET 去重，TTL 10min）

3. **防刷机制**：
   - 接口限流（Sentinel）：单用户每分钟最多贡献 N 次热度
   - 异常检测：某关键词短时间热度飙升异常（如 1 分钟涨 10 倍），标记为疑似刷榜，人工审核

4. **分时段**：
   - `hot_search:1h`、`hot_search:24h`、`hot_search:week` 三个 ZSET
   - 1h 榜每分钟从实时 ZSET 截取 Top 50
   - 24h 榜每小时从 1h 榜聚合
   - week 榜每天从 24h 榜聚合

5. **展示层**：
   - Gateway → Search 服务 → 读 Redis ZSET Top 50 → 返回
   - 加本地缓存（Caffeine，TTL 1min），避免每次请求都查 Redis

### Q36：如果小红书突然爆火，某个笔记瞬间 10 万并发访问，你的系统会怎样？怎么抗住？

**答：**

**当前系统瓶颈分析**：
1. **Gateway**：Spring Cloud Gateway 基于 Netty，非阻塞 IO，单实例可扛数万并发，但需要多实例 + SLB
2. **Note 服务**：三级缓存 Caffeine(10K) → Redis → MySQL，热点笔记会命中 Caffeine，但 10 万并发可能打满 Tomcat 线程池（默认 200）
3. **Redis**：单节点 QPS 约 10 万，10 万并发 GET 可能扛住，但如果有大量 miss 穿透到 DB，Redis 也扛不住
4. **MySQL**：单实例 QPS 约 5000-10000，10 万并发肯定扛不住

**抗住方案**：

1. **热点探测 + 本地缓存预热**：
   - 运行时识别热点 key（如 QPS > 1000），主动推送到所有实例的 Caffeine 本地缓存
   - 本地缓存命中率 99%+，请求不落 Redis

2. **多级缓存扩容**：
   - Caffeine 容量从 10K 扩大到 100K
   - Redis Cluster 扩节点

3. **限流降级**：
   - Gateway 层 Sentinel 限流：单笔记 QPS 超 5 万时排队/拒绝
   - Note 服务降级：超载时返回简化版详情（不含 KV 内容，只有标题+图片）

4. **CDN 静态化**：
   - 笔记详情页 SSR（服务端渲染）后缓存到 CDN
   - 图片/视频已在 OSS + CDN，无压力

5. **DB 读写分离**：
   - 读请求走从库，写请求走主库
   - 热点笔记详情几乎都是读请求

6. **弹性伸缩**：
   - K8s HPA 根据 CPU/QPS 自动扩容 Note 服务 Pod

### Q37：点赞功能从 Redis 异步落 DB，如果 Redis 和 DB 的计数不一致，用户看到的是哪个？怎么让用户感知一致？数据对齐的延迟是多少？

**答：**

**用户看到的是 Redis 的值**：
- 前端展示点赞数时，调用 Count 服务，Count 服务优先读 Redis Hash
- Redis 是实时更新的，DB 是异步落盘的，两者可能短暂不一致

**用户感知一致性问题**：
- **场景**：用户 A 点赞后，刷新页面看到点赞数 +1（Redis 已更新），但 DB 还没落盘
- **问题**：如果 Redis 数据丢失（如宕机），从 DB 恢复后点赞数回退，用户困惑
- **解决**：
  1. **前端乐观更新**：点赞后前端立即 +1，不等后端返回，避免延迟感知
  2. **数据对齐兜底**：XXL-JOB 每天对齐，确保 DB 最终和 Redis 一致
  3. **Redis 持久化**：开启 AOF，减少数据丢失概率
  4. **关键操作同步落 DB**：对一致性要求极高的操作（如支付），同步写 DB + Redis

**数据对齐延迟**：
- 日常：MQ 异步落盘，延迟约 **100ms - 1s**
- 异常恢复：XXL-JOB 每天对齐，最大延迟 **24 小时**
- 可以增加对齐频率（每小时），但会增加 DB 压力

### Q38：你这个系统里哪些数据是最终一致的？哪些是强一致的？最终一致的性能收益有多大？如果产品要求点赞数必须实时准确，你怎么改？代价是什么？

**答：**

**最终一致的数据**：
- 点赞/取消点赞状态（Redis → MQ → DB）
- 收藏/取消收藏状态
- 关注/取关状态
- 计数数据（点赞数、收藏数、评论数、粉丝数）
- 评论热度
- ES 搜索索引（Canal binlog 同步）

**强一致的数据**：
- 用户注册（事务写入 MySQL）
- 笔记发布（RocketMQ 事务消息保证元数据 + KV 内容一致）
- 用户资料更新（直接写 MySQL，删缓存）

**最终一致的性能收益**：
- 以点赞为例：
  - 强一致方案：写 DB + 更新 Redis + 返回，耗时 ≈ DB 写入 5ms + Redis 更新 1ms = **6ms**
  - 最终一致方案：写 Redis + 发 MQ + 返回，耗时 ≈ Redis 写入 1ms + MQ 异步发送 0.5ms = **1.5ms**
  - **性能提升 4 倍**，且 DB 不成为瓶颈

**如果要求实时准确**：
- 改为同步写 DB + Redis（双写），或写 DB → 事务成功后更新 Redis
- 代价：
  1. **延迟增加**：每次点赞要等 DB 写入完成，RT 从 1.5ms → 6ms
  2. **吞吐下降**：DB 成为瓶颈，QPS 从 10 万+ → 5000-10000
  3. **DB 压力**：高并发写入可能打挂 DB
  4. **需要分库分表**：单表扛不住，增加架构复杂度

### Q39：如果让你做全链路压测，你会怎么设计？哪些是瓶颈点？

**答：**

**全链路压测设计**：

1. **压测场景**：
   - 笔记详情读（最高频）：Gateway → Note → User(Feign) → KV(Feign) → Count(Feign)
   - 首页 Feed 流：Gateway → Recommend → ES → Note → User → Count
   - 点赞写：Gateway → Note → Redis → MQ
   - 搜索：Gateway → Search → ES

2. **压测工具**：JMeter 或 Gatling，模拟真实用户行为（思考时间、浏览→点赞→收藏路径）

3. **数据准备**：
   - 10 万用户、100 万笔记、1000 万点赞/收藏
   - Redis 预热热点数据
   - ES 预建索引

4. **瓶颈点分析**：

   | 环节 | 瓶颈原因 | 预估 QPS |
   |------|---------|---------|
   | Gateway | CPU 密集（Sa-Token 鉴权 + 路由匹配） | 3-5 万/实例 |
   | Note 服务 | Tomcat 线程池（200）、Feign 调用链路长 | 5000-10000/实例 |
   | Redis | 单节点 10 万 QPS，热点 key 集中 | 10 万/节点 |
   | MySQL | 磁盘 IO、锁争用、连接池 | 5000-10000/实例 |
   | ES | 堆内存、Refresh 频率、Segment Merge | 5000-10000/实例 |
   | Cassandra | LSM Tree Compaction、读放大 | 1-3 万/实例 |
   | Feign | HTTP 连接池、序列化开销 | 取决于下游 |

5. **优化优先级**：
   - P0：Note 服务加本地缓存预热 + 扩实例
   - P1：Redis Cluster 扩节点 + 读写分离
   - P2：MySQL 读写分离 + 热点查询优化
   - P3：Gateway 多实例 + SLB

### Q40：你用了策略模式切换 OSS 存储，@RefreshScope 动态刷新，@RefreshScope 的原理？如果切换过程中有文件正在上传怎么办？MinIO 和 OSS 的 SDK 差异怎么屏蔽的？

**答：**

**@RefreshScope 原理**：
1. Spring 容器启动时，`@RefreshScope` 标注的 Bean 不会立即创建，而是创建一个**代理对象（CGLIB 代理）**
2. 首次访问代理时，从 **Scope 缓存** 中获取或创建真实 Bean
3. 当 Nacos 配置变更触发 `/actuator/refresh` 时，Spring Cloud 发出 `RefreshEvent`
4. `RefreshScope` 监听到事件后，**销毁缓存中的 Bean 实例**
5. 下次访问代理时，重新创建 Bean（使用最新配置），实现动态刷新

**切换过程中文件上传问题**：
- 场景：用户上传文件，Strategy 选择 MinIO → 配置变更切换为 OSS → 上传中途 Strategy 变了
- **当前风险**：如果 Strategy 是无状态的（只依赖配置），切换后新请求用新 Strategy，正在进行的请求仍用旧 Strategy（因为 Bean 已在方法栈中），所以**不会出问题**
- 但如果旧 Strategy 持有的连接池/客户端被销毁，正在进行的请求可能抛异常
- **解决方案**：
  1. **优雅切换**：新 Strategy 就绪后，旧 Strategy 等待进行中的请求完成再销毁（引用计数 + 优雅停机）
  2. **重试机制**：上传失败后用新 Strategy 重试
  3. **双写过渡**：切换期间同时写两个存储，确认新存储可用后切读

**SDK 差异屏蔽**：
- `FileStrategy` 接口定义统一方法：`uploadFile(MultipartFile)`、`deleteFile(String fileUrl)`
- `MinioFileStrategy` 和 `AliyunOSSFileStrategy` 各自实现，内部封装 SDK 差异
- 控制层只依赖 `FileStrategy` 接口，不感知底层实现
- `FileStrategyFactory` 根据 Nacos 配置 `storage.type` 选择实现类，`@RefreshScope` 保证配置变更时重新选择
