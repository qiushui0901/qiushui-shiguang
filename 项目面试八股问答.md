# 小哈书项目面试八股问答

## 背诵原则

回答时不要只背概念，建议按这个结构说：

1. 先说项目场景：这个技术在项目里用在哪里。
2. 再说八股原理：它解决什么通用问题。
3. 最后说取舍：为什么这么设计，有什么代价，怎么兜底。

示例：

> 点赞这里用了 Redis Lua，是因为点赞和判断是否已点赞必须原子化，否则并发下可能重复点赞。Lua 可以把判断、写入集合、返回状态放在 Redis 单线程里执行。后面再通过 RocketMQ 异步同步 DB 和计数，所以这里牺牲了强一致性，换取接口低延迟和高吞吐。

## 一、项目整体

### 1. 这个项目的核心业务链路是什么？

这个项目是一个仿小红书的内容社区后端，核心链路是用户登录注册、发布笔记、上传图片或视频、浏览笔记详情、点赞收藏、评论、关注用户、搜索笔记、推荐内容。

从架构上看，用户请求先经过 Gateway 网关，网关做鉴权和路由，然后转发到认证、用户、笔记、评论、计数、搜索、推荐等微服务。高频写操作，比如点赞、收藏、评论、关注，会先写 Redis 或发送 MQ，再异步落库和更新计数。

### 2. 为什么要拆成这么多微服务？

因为不同业务的访问特点和扩展需求不同。比如认证、用户、笔记、评论、计数、搜索、推荐的职责差异很明显，拆开以后可以独立开发、独立部署、独立扩容。

举例来说，评论和点赞是高并发写，搜索依赖 Elasticsearch，KV 服务依赖 Cassandra，OSS 服务依赖对象存储。如果都放在一个服务里，耦合会比较重。拆成微服务后，笔记服务可以专注笔记元数据，正文交给 KV，计数交给 count，搜索交给 search。

代价是系统复杂度上升，需要服务注册发现、网关、配置中心、链路追踪、容错降级和分布式一致性设计。

### 3. api 模块和 biz 模块为什么要分开？

`api` 模块主要放 Feign 接口、DTO、常量等对外契约，`biz` 模块放具体业务实现。

这样其他服务只依赖 `api`，不用依赖对方的完整实现，能降低耦合。比如笔记服务需要调用用户服务，只需要依赖 `shiguang-user-api` 里的 Feign 接口和 DTO。

这种拆法的好处是服务边界清楚，坏处是 DTO 版本管理要谨慎，否则接口字段变更会影响多个服务。

### 4. 网关在项目里做了什么？

网关主要做统一入口、路由转发、认证鉴权和用户信息透传。请求进入网关后，根据路径转发到不同服务，比如 `/note/**` 转到笔记服务，`/comment/**` 转到评论服务。

同时网关会校验 Sa-Token 登录态，从 Redis 中解析 token 对应的用户 ID，然后把用户 ID 放到请求头里传给下游服务。下游通过上下文工具类获取当前登录用户。

这样认证逻辑集中在网关，业务服务不用重复做登录校验。

## 二、微服务与 Spring Cloud

### 5. Nacos 在项目里有什么作用？

Nacos 主要承担服务注册发现和配置管理。每个微服务启动后会注册到 Nacos，网关和 Feign 调用可以通过服务名找到对应实例。

比如网关配置的是 `lb://shiguang-note`，不是固定 IP，说明它通过注册中心做负载均衡。这样服务扩容、实例迁移时，调用方不用改配置。

### 6. 为什么用 OpenFeign？

OpenFeign 是声明式 HTTP 客户端，适合微服务之间调用。相比手写 RestTemplate，它只需要定义接口和注解，就能像调用本地方法一样调用远程服务。

在这个项目里，用户、计数、KV、OSS、分布式 ID 都提供了 Feign API。比如笔记服务发布笔记时会调用分布式 ID 服务生成 ID，查询详情时会调用 KV 服务拿正文。

但 Feign 本质还是远程调用，会有超时、失败、网络抖动问题，所以必须配置超时、重试、熔断和 fallback。

### 7. Feign fallback 直接 return null 有什么问题？

直接 `return null` 风险比较大，因为调用方可能无法区分是“真的没数据”还是“远程服务调用失败”。如果上层没有判空，就会出现 NPE；如果上层把 null 当正常结果处理，还可能产生错误业务数据。

更好的做法是返回明确的失败响应，比如 `Response.fail()`，或者抛出业务异常，让上层走降级逻辑。对于核心链路，比如发笔记需要 ID 服务，如果 ID 服务不可用，应该快速失败，而不是返回 null。

### 8. Sentinel 限流、熔断、降级分别是什么？

限流是限制单位时间内的请求量，防止服务被打爆。熔断是当下游服务错误率或响应时间过高时，暂时切断调用，避免故障扩散。降级是在服务不可用时返回兜底结果，比如返回默认值、空列表或友好提示。

在这个项目里，高并发入口比如笔记详情、评论列表、计数查询、搜索接口都适合做限流。Feign 调用用户、计数、KV 服务时，也适合做熔断和降级。

## 三、网关与认证

### 9. 为什么认证逻辑放在网关？

因为网关是所有外部请求的统一入口，把认证放在网关可以避免每个业务服务重复写登录校验逻辑，也能提前拦截非法请求，减少下游压力。

网关认证通过后，会把用户 ID 写入请求头传给下游服务。下游服务只负责业务处理，从上下文中获取当前用户 ID。

不过要注意，下游服务不能信任外部直接传入的用户 ID 请求头，必须保证业务服务只允许网关访问，或者在内网层面做隔离。

### 10. Sa-Token 登录态是怎么维护的？

用户登录成功后，认证服务调用 Sa-Token 生成 token，并把 token 和用户 ID 的映射存到 Redis。客户端后续请求携带 `Authorization: Bearer token`，网关根据 token 去 Redis 查询登录态。

它本质上更接近服务端 session token，而不是纯 JWT。好处是服务端可以主动踢人、续期、失效 token；缺点是依赖 Redis。

### 11. JWT 和 Session Token 有什么区别？

JWT 是自包含 token，用户信息和签名都在 token 里，服务端一般不需要查 Redis，适合无状态认证。缺点是签发后不好主动失效，token 泄露后风险较大。

Session Token 只是一串随机值，状态存在服务端，比如 Redis。好处是可以主动删除、续期、控制登录状态；缺点是每次校验都依赖存储。

这个项目用 Sa-Token + Redis，适合需要服务端统一管理登录态和权限的场景。

## 四、Redis 缓存

### 12. 项目里 Redis 用在哪些地方？

Redis 用得比较多，主要包括登录 token、权限缓存、笔记详情缓存、用户信息缓存、评论列表 ZSET、关注/粉丝列表 ZSET、点赞收藏状态、计数缓存、数据对齐增量记录、Agent 会话记忆。

整体上，Redis 在项目里承担了高频读缓存、高频写缓冲、集合关系存储和临时状态存储的作用。

### 13. 为什么笔记详情要用 Caffeine + Redis + DB？

这是典型的多级缓存。Caffeine 是本地缓存，访问最快，适合缓存热点数据；Redis 是分布式缓存，多实例共享；DB 是最终数据源。

查询笔记详情时，先查本地缓存，再查 Redis，最后查 DB 和 KV。这样可以减少 Redis 和数据库压力，提高热点数据访问速度。

代价是多实例本地缓存会有一致性问题，所以项目里更新或删除笔记时，会通过 MQ 通知各实例删除本地缓存，并配合 Redis 延迟双删降低脏数据概率。

### 14. 缓存穿透、击穿、雪崩分别是什么？

缓存穿透是查询不存在的数据，请求每次都打到数据库。解决方式是缓存空值、参数校验、布隆过滤器。

缓存击穿是某个热点 key 过期，大量请求同时打到数据库。解决方式是互斥锁、逻辑过期、热点 key 不过期。

缓存雪崩是大量 key 同时过期或 Redis 故障，导致请求集中打到数据库。解决方式是过期时间加随机值、Redis 高可用、限流降级。

这个项目里用本地缓存、Redis、随机过期时间、延迟双删等方式缓解缓存问题。

### 15. 延迟双删是什么？为什么要用？

延迟双删是更新数据库前后删除缓存，或者更新数据库后先删一次缓存，再延迟一段时间删第二次缓存。

它解决的是并发读写下缓存脏数据的问题。比如线程 A 更新 DB 并删除缓存时，线程 B 可能刚好读到旧数据并重新写入缓存。延迟再删一次，可以把这种旧缓存清掉。

这个方案不能保证绝对强一致，但实现简单，适合允许短暂不一致的读多写少场景。

### 16. Redis ZSET 为什么适合评论列表和关注列表？

ZSET 同时有成员和分数，可以按分数排序。评论列表可以用热度或时间作为 score，关注/粉丝列表可以用关注时间作为 score。

查询时可以用 `reverseRangeByScore` 做倒序分页，天然适合“按时间倒序”或“按热度倒序”的列表。

但 ZSET 不适合无限存储全量冷数据，所以项目里通常只缓存前几百条热点数据，深分页还是回源数据库。

### 17. Redis Lua 脚本解决了什么问题？

Lua 脚本可以把多个 Redis 操作打包成一个原子操作执行。Redis 执行 Lua 时是单线程串行的，不会被其他命令打断。

在点赞、收藏、关注场景中，需要先判断是否已经操作过，再决定新增或删除集合成员。如果用多条 Redis 命令，存在并发竞态；用 Lua 可以保证判断和写入的原子性。

所以项目里高并发状态变更适合用 Lua，比如点赞幂等、取消点赞、关注、取关。

## 五、RocketMQ

### 18. 项目里哪些业务用了 MQ？

笔记发布、点赞收藏、评论发布、评论删除、关注取关、计数更新、缓存删除、数据对齐、推荐兴趣更新等都用了 RocketMQ。

总体思路是：用户请求先快速完成关键状态变更，然后通过 MQ 异步处理非核心或耗时操作，比如落库、更新计数、删除缓存、同步推荐画像。

### 19. 点赞、收藏、评论为什么不直接同步写 DB？

因为这些是高频操作，如果每次都同步写 MySQL，会导致数据库压力很大，接口响应也会变慢。

项目里一般先写 Redis，保证用户马上看到状态变化，然后发 MQ 异步写 DB 和更新计数。这种设计牺牲了短时间内的强一致性，换取高吞吐和低延迟。

这种场景适合最终一致性，因为点赞数短暂延迟用户通常可以接受。

### 20. RocketMQ 顺序消息解决什么问题？

顺序消息保证同一个业务 key 的消息按顺序消费。比如同一个用户对同一篇笔记先点赞后取消点赞，如果消费顺序反了，最终状态就会错。

项目里点赞、收藏、关注这类有正反操作的业务，适合用业务 ID 作为 hashKey 发送顺序消息，保证同一实体的操作顺序。

### 21. 事务消息解决什么问题？

事务消息用来保证本地事务和 MQ 消息发送的一致性。比如发布笔记时，既要写笔记元数据，又要保存正文内容或通知其他服务。如果 DB 写成功但 MQ 没发出去，后续链路就断了。

RocketMQ 事务消息先发送半消息，再执行本地事务，最后提交或回滚消息。如果状态不确定，Broker 会回查本地事务状态。

在项目里，笔记发布正文非空时使用事务消息，目的是保证笔记发布和正文保存/后续处理链路的一致性。

### 22. MQ 重复消费怎么办？

MQ 默认不能保证绝对只消费一次，所以消费者必须做幂等。

常见方式有：数据库唯一索引、业务状态判断、Redis 去重 key、版本号、操作流水表。比如点赞落库可以用用户 ID + 笔记 ID 做唯一索引，重复插入直接忽略；计数消息可以带操作类型，消费时按当前状态判断。

一句话：MQ 只能帮我们尽量可靠投递，业务层必须保证幂等。

### 23. 如果 MQ 挂了，点赞接口还能成功吗？

要看设计目标。如果点赞接口先写 Redis，再发 MQ，那么 Redis 写成功但 MQ 失败会导致用户看到点赞成功，但 DB 和计数可能不同步。

更稳妥的做法是 MQ 发送失败时记录补偿任务，或者将待发送消息写入本地消息表，再由定时任务重试。对于强依赖 MQ 的链路，也可以直接返回失败。

这个项目偏最终一致性设计，应该补充失败重试和补偿机制。

## 六、MySQL 与事务

### 24. 为什么评论总数不直接 count(*)？

因为评论表数据量大时，`count(*)` 会带来较大数据库压力，尤其是高频查询笔记详情或评论列表时。

项目里把评论数、点赞数、收藏数放到计数表或 Redis 计数中，查询时直接读计数结果，避免每次扫评论表。

这是一种典型的空间换时间设计，代价是要维护计数一致性。

### 25. 深分页为什么慢？怎么优化？

MySQL 深分页慢是因为 `limit offset, size` 需要先扫描并丢弃 offset 条数据，offset 越大越慢。

优化方式有：基于游标分页，比如 `where id < lastId order by id desc limit size`；使用覆盖索引；冷热数据分离；只允许查询前 N 页；或者把热点列表放 Redis ZSET。

项目里的评论列表就适合前几百条走 Redis ZSET，深分页回源 DB。

### 26. 事务传播机制是什么？项目哪里需要事务？

事务传播机制指一个事务方法调用另一个事务方法时，事务如何传播。常见的是 `REQUIRED`，如果当前有事务就加入，没有就新建。

项目里需要事务的地方包括：发布笔记本地入库、评论入库并更新评论关系、点赞收藏落库、关注取关落库、计数落库等。

但分布式场景下，本地事务只能保证单服务单数据库一致性，跨服务一致性通常要靠 MQ 事务消息、补偿任务和幂等消费。

## 七、分布式 ID

### 27. 为什么不用数据库自增 ID？

数据库自增 ID 在单库场景简单好用，但在分库分表和微服务场景下会有瓶颈。多个服务或多个库同时生成 ID 时，容易冲突，也依赖数据库中心节点。

这个项目有用户、笔记、评论等多个服务，需要全局唯一 ID，所以使用分布式 ID 服务。

### 28. Snowflake 雪花算法是什么？

Snowflake 通常由时间戳、机器 ID、序列号组成。它可以在本地生成趋势递增的全局唯一 ID，不需要每次访问数据库。

优点是性能高、可排序、去中心化。缺点是依赖机器时钟，如果发生时钟回拨，可能生成重复 ID。

### 29. Leaf Segment 和 Snowflake 有什么区别？

Leaf Segment 是号段模式，服务先从数据库申请一段 ID，比如 1 到 10000，然后在内存中递增使用。它对时钟不敏感，但依赖数据库发号段。

Snowflake 是本地按时间生成 ID，性能更高，但依赖机器时钟和 workerId 分配。

简单说：Segment 更稳定依赖 DB，Snowflake 更高性能依赖时钟。

## 八、Elasticsearch 搜索

### 30. 为什么不用 MySQL LIKE 做搜索？

MySQL `LIKE '%关键词%'` 很难走普通索引，数据量大时性能差，而且相关性排序、分词、拼音、热度排序都不好做。

Elasticsearch 基于倒排索引，适合全文检索，可以按标题、话题、内容分词搜索，也支持相关性评分、高亮、聚合和复杂排序。

所以笔记搜索、用户搜索、搜索建议更适合用 ES。

### 31. 倒排索引是什么？

倒排索引是从“词”到“文档”的映射。比如“露营”这个词出现在哪些笔记里，ES 会建立对应关系。

用户搜索关键词时，ES 不需要扫描所有笔记，而是直接根据词找到相关文档，再计算相关性得分。

### 32. MySQL 数据如何同步到 ES？

项目里使用 Canal 监听 MySQL binlog，当笔记或用户数据变化时，Canal 捕获变更，然后同步到 Elasticsearch。

这样可以让 MySQL 作为主数据源，ES 作为搜索索引。缺点是同步存在延迟，刚发布的内容可能短时间内搜不到。

解决方式可以是发布成功后主动写 ES，或者搜索不到时回源 MySQL，或者接受短暂最终一致性。

### 33. 笔记搜索如何按相关性、热度、时间综合排序？

可以先用关键词匹配标题、话题、正文得到相关性分数，再把点赞数、收藏数、评论数作为热度因子，最后加时间衰减，让新内容有一定曝光。

项目 Agent 搜索总结里就有类似思路：标题和话题匹配，再结合 like、collect、comment 和时间衰减做综合评分。

这比单纯按关键词匹配更符合内容社区的搜索体验。

## 九、评论系统

### 34. 一级评论和二级评论怎么建模？

评论表可以有 `comment_id`、`note_id`、`parent_id` 或 `reply_comment_id`、`creator_id`、`content_uuid`、`status`、`create_time` 等字段。

一级评论的父评论为空，二级评论关联一级评论或被回复评论。查询时通常先查一级评论列表，再批量查每个一级评论下的部分二级评论。

为了性能，评论正文可以放 KV，评论元数据放 MySQL，热点评论放 Redis。

### 35. 热门评论怎么排序？

热门评论可以用热度分排序，热度分一般由点赞数、回复数、时间衰减组成。比如点赞越多分越高，越新的评论适当加权。

Redis ZSET 很适合存评论热度，member 是评论 ID，score 是热度分。查询评论列表时按 score 倒序取出评论 ID，再批量查详情。

### 36. 评论删除后子评论怎么处理？

有两种方式：物理删除和逻辑删除。内容社区一般更适合逻辑删除，把评论状态改为删除，这样可以保留评论树结构，避免子评论关系断裂。

如果删除一级评论，可以选择隐藏所有子评论，或者展示“该评论已删除”但保留子评论。具体取决于产品规则。

项目里还需要同步删除 Redis 缓存、本地缓存，并通过 MQ 更新计数和热度。

## 十、计数系统

### 37. 为什么单独做 count 服务？

点赞数、收藏数、评论数、粉丝数、关注数都是高频读写数据。如果每个业务服务都自己维护计数，会造成重复逻辑和数据不一致。

单独做 count 服务，可以统一维护计数表和计数缓存。笔记详情、用户主页、评论列表都可以调用 count 服务查询计数。

代价是多了一次远程调用，所以要配合缓存和 fallback。

### 38. 计数服务如何避免高频写 MySQL？

高频计数先写 Redis，或者通过 MQ 聚合后批量写 MySQL。比如点赞消息先更新 Redis 计数，再异步发送到计数落库消费者。

还可以用 buffer-trigger 或定时任务，把多次增量合并成一次 DB 更新。

这属于最终一致性设计，适合点赞数、收藏数这种可以短暂延迟的数据。

### 39. Redis 计数丢失怎么办？

Redis 计数丢失可能来自 Redis 宕机、缓存淘汰、数据过期、误删。解决方式包括开启 Redis 持久化、主从高可用、定时从 DB 对账、把 MQ 消息作为补偿来源。

项目里的 `data-align` 服务就是类似数据对齐模块，通过定时任务把 Redis 中的增量数据同步或校准到 DB。

## 十一、OSS 与 KV

### 40. 为什么笔记正文不直接放 MySQL？

笔记正文和评论正文属于较长文本，如果直接放 MySQL 主表，会让主表变宽，影响查询性能。笔记列表和详情常用的是标题、图片、作者、计数等元数据，不一定每次都需要正文。

所以项目把结构化元数据放 MySQL，把正文放 KV 服务，比如 Cassandra。这样可以减少 MySQL 压力，也方便正文独立扩展。

代价是详情查询需要多一次 KV 调用，并且要处理元数据和正文的一致性。

### 41. 图片为什么走 OSS/MinIO，不存数据库？

图片和视频是大对象，不适合存数据库。数据库更适合存结构化数据，文件应该存对象存储，比如 MinIO 或阿里云 OSS。

业务表只存文件 URL 或 object key。这样可以减少数据库体积，文件访问也可以走 CDN。

上传时还要校验文件大小、类型、后缀、内容安全，防止恶意文件。

## 十二、推荐系统

### 42. 推荐服务的数据来源是什么？

推荐服务的数据主要来自用户行为，比如点赞、收藏、评论、关注、搜索、浏览。项目里推荐服务会消费点赞和收藏 MQ，维护用户兴趣画像。

推荐时可以根据用户兴趣标签召回相关笔记，再结合热度、时间、作者关系等因素排序。

### 43. 冷启动用户怎么推荐？

冷启动用户没有历史行为，可以用热门内容、最新内容、运营精选、用户注册时选择的兴趣标签、地理位置或相似用户行为来推荐。

随着用户产生点赞、收藏、搜索等行为，再逐步切换到个性化推荐。

### 44. 点赞和收藏的兴趣权重一样吗？

不应该完全一样。收藏通常表示更强兴趣，因为用户愿意保存内容；点赞可能是轻量反馈。可以设置收藏权重大于点赞，比如收藏权重 3，点赞权重 1。

评论、关注作者、停留时长也可以作为更强的兴趣信号。

## 十三、AI Agent

### 45. Agent 在这个项目里解决什么问题？

Agent 不是核心业务必需模块，而是 AI 增强能力。基础业务，比如登录、发笔记、点赞、评论、关注、搜索、推荐，都可以不依赖 Agent。

Agent 的价值是把搜索、推荐、内容审核、创作辅助等能力包装成工具，通过自然语言入口统一编排。比如用户说“帮我找几篇适合新手露营的笔记并总结重点”，Agent 可以调用搜索工具、整理结果、生成总结。

### 46. Agent 和普通调用大模型有什么区别？

普通调用大模型通常是输入一段 prompt，输出一段文本。Agent 更强调任务规划、工具调用和多步骤执行。

在项目里，Agent 可以先做内容理解，再做内容审核，然后调用搜索或推荐工具，最后由创作助手生成结果。它不是单纯聊天，而是把大模型和后端工具连接起来。

### 47. 为什么要把搜索、推荐包装成 Tool？

因为大模型本身不知道项目数据库里的实时内容，也不能直接访问业务服务。把搜索、推荐包装成 Tool 后，大模型可以在需要时调用这些工具，拿到真实业务数据，再生成回答。

这样既能利用大模型的自然语言能力，又能保证结果基于项目已有数据，而不是纯靠模型编造。

### 48. 多 Agent 串行和并行有什么区别？

串行是一个 Agent 的输出作为下一个 Agent 的输入，适合有依赖关系的任务，比如先审核，再创作。

并行是多个 Agent 同时执行，适合互不依赖的任务，比如同时做内容理解和风险判断，最后聚合结果。并行速度更快，但上下文协调更复杂。

### 49. Agent 输出不稳定怎么办？

要做约束和兜底。比如使用固定 JSON 输出格式、限制工具权限、增加内容审核、对关键结果做业务校验、失败时返回普通搜索结果。

对于核心业务，不能让 Agent 直接决定数据库写操作。Agent 更适合做辅助推荐、总结、创作建议。涉及真实写操作时，必须经过明确的业务接口和权限校验。

### 50. SSE 流式输出原理是什么？

SSE 是 Server-Sent Events，服务端通过一个长连接持续向浏览器推送文本事件。客户端可以边接收边展示，适合大模型流式输出。

相比普通 HTTP 一次性返回，SSE 可以降低用户感知延迟。相比 WebSocket，SSE 更简单，适合服务端单向推送。

## 十四、高并发与一致性

### 51. 点赞接口 QPS 很高，你会怎么抗？

首先不直接写 MySQL，而是用 Redis 存点赞状态，用 Lua 保证原子和幂等。然后通过 RocketMQ 异步落库和更新计数。

其次对热点 key 做本地缓存或分片处理，对接口做限流，对 MQ 消费做水平扩容。数据库侧用唯一索引保证最终幂等。

整体思路是 Redis 扛实时状态，MQ 削峰，MySQL 保存最终结果。

### 52. Redis 和 DB 数据不一致怎么办？

要先明确一致性要求。点赞数、评论数这类数据可以接受最终一致，通过 MQ、重试、定时对账修复。

对于关键数据，比如用户账号、笔记是否存在，应该以 DB 为准，缓存只做加速。更新时先更新 DB，再删除缓存，并配合延迟双删或消息通知删除本地缓存。

### 53. 本地缓存多实例不一致怎么办？

本地缓存是每个服务实例独立的，一个实例更新了数据，其他实例的本地缓存不会自动失效。

解决方式是通过 MQ 或 Redis Pub/Sub 广播缓存删除消息，所有实例收到消息后删除本地缓存。也可以缩短本地缓存过期时间，降低不一致窗口。

项目里笔记详情、评论详情都有本地缓存，更新或删除时需要发 MQ 删除本地缓存。

### 54. 如何设计接口幂等？

幂等就是同一个请求执行多次，结果和执行一次一致。

常见做法有：唯一索引、防重 token、业务状态判断、Redis setnx、请求流水号。比如点赞可以用用户 ID + 笔记 ID 唯一索引；评论发布可以用客户端请求 ID 防重复提交；MQ 消费可以用消息 ID 或业务唯一键去重。

在分布式系统里，接口和消费者都要考虑幂等。

### 55. 如何做限流？

限流可以在网关、服务接口、Redis、Sentinel 层做。常见算法有固定窗口、滑动窗口、漏桶、令牌桶。

项目里适合在网关对登录、发评论、点赞、搜索等接口做限流，也可以用 Sentinel 对服务级别的 QPS 和 RT 做保护。

限流后要返回明确提示，不能直接让系统异常。

## 十五、项目风险与优化

### 56. 项目里配置了明文密码，生产怎么处理？

生产环境不能把数据库、Redis、OSS、API Key 明文写在代码仓库里。应该放到环境变量、Nacos 配置中心、KMS 或密钥管理系统中。

配置文件里只保留占位符，比如 `${MYSQL_PASSWORD}`。同时要区分 dev、test、prod 环境，避免开发配置误上生产。

### 57. NoteServiceImpl 很大，你会怎么重构？

可以按职责拆分。比如拆成笔记发布服务、笔记查询服务、点赞收藏服务、缓存服务、MQ 发布器、Lua 执行器、RPC 网关。

这样每个类只负责一类事情，单元测试也更容易写。比如点赞收藏的 Lua 和 MQ 逻辑可以独立成 `NoteInteractionService`，笔记详情缓存可以独立成 `NoteCacheService`。

重构时不能一次性大改，应该先抽无状态工具和基础组件，再逐步迁移业务方法。

### 58. 如果 MQ 消费积压，你怎么排查？

先看积压的是哪个 Topic 和 Consumer Group，再看生产速度、消费速度、消费者实例数、单条消息处理耗时、是否有异常重试。

常见原因有消费者处理太慢、下游 DB 慢、某条毒消息反复失败、分区不均、消费者线程数太少。

解决方式包括扩容消费者、优化消费逻辑、批量写 DB、跳过或隔离毒消息、提高线程数、拆分 Topic。

### 59. 如果接口偶发慢，你怎么定位？

先看监控和日志，确认慢在网关、业务服务、Redis、DB、MQ 还是远程 Feign 调用。

具体可以查接口 RT、线程池队列、慢 SQL、Redis 慢命令、GC 日志、连接池使用率、下游服务耗时。对于微服务项目，最好接入链路追踪，能看到每一段调用耗时。

定位思路是先判断范围，再逐层缩小。

## 高频总结

这个项目最容易被深挖的点：

1. Redis：多级缓存、Lua、ZSET、缓存穿透/击穿/雪崩、延迟双删、本地缓存一致性。
2. MQ：异步削峰、事务消息、顺序消息、重复消费、幂等、最终一致性。
3. 微服务：Gateway、Nacos、Feign、Sa-Token、Sentinel。
4. MySQL：索引、事务、深分页、计数表、分库分表。
5. ES：倒排索引、分词、相关性排序、Canal 同步。
6. 分布式 ID：Snowflake、Leaf Segment、时钟回拨。
7. Agent：工具调用、多 Agent 编排、SSE、模型输出兜底。

## 万能回答模板

### 技术选型模板

> 我们在这个场景选择 X，是因为这里的核心矛盾是高并发读写和响应延迟。X 能解决某个关键问题，比如 Redis 解决热点访问，MQ 解决异步削峰，ES 解决全文检索。它的代价是引入一致性或运维复杂度，所以项目里又通过幂等、重试、对账、降级来兜底。

### 一致性模板

> 这个地方没有追求强一致，而是最终一致。因为点赞数、收藏数、评论数这类数据允许短暂延迟。接口先写 Redis 保证用户体验，再通过 MQ 异步落库，消费者做幂等，失败后重试，最后通过定时对账修复异常数据。

### 缓存模板

> 这个接口是典型读多写少，所以用了多级缓存。先查本地 Caffeine，再查 Redis，最后查 DB。更新时不直接更新缓存，而是删除缓存，并用延迟双删和 MQ 删除本地缓存，降低并发场景下的脏读概率。

### MQ 模板

> 这个业务用 MQ 是为了异步削峰和服务解耦。用户请求只完成核心状态变更，耗时的落库、计数、缓存删除、推荐画像更新都通过 MQ 异步处理。因为 MQ 可能重复投递，所以消费者必须保证幂等。

### Agent 模板

> Agent 不是基础业务必需模块，而是 AI 增强能力。它的价值是把搜索、推荐、审核、创作等后端能力包装成工具，然后由大模型根据用户自然语言进行多步骤编排。代价是成本、延迟和不确定性，所以不能让它直接影响核心业务链路，必须做权限控制、输出校验和降级兜底。
